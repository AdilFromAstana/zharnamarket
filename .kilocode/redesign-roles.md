# 🔄 Пересмотр системы ролей — от «активной роли» к «действиям по запросу»

**Дата:** 25.03.2026  
**Статус:** Утверждено к реализации  
**Причина:** Текущая система с переключением ролей создаёт ненужный барьер для пользователей

---

## 🧠 Проблема старой системы

### Что было

Пользователь при регистрации **обязан** выбрать роль:
- 👤 Я креатор
- 💼 Я бизнес  
- 🔄 Оба

В интерфейсе есть `activeRole` — «активный режим», который нужно переключать. Кабинет зависел от активной роли: креаторы шли на `/profile`, бизнес — на `/business/ads`.

### Почему это плохо

Владелец кофейни (bakery) заходит на платформу с одной мыслью: **разместить объявление**. Вместо этого он видит:
1. «Выберите роль» — непонятно зачем
2. «Активируйте режим бизнеса» — лишний шаг
3. Кнопку «Переключить на бизнес» в шапке — непонятно когда нужна

Это имитация сложных платформ (Upwork, Airbnb), где роли критичны. Но мы — **платная доска объявлений**. На OLX не спрашивают «ты покупатель или продавец?» — просто заходишь и делаешь нужное действие.

---

## ✅ Новая концепция: «Действия по запросу»

### Ключевой принцип

**Нет ролей. Есть действия.** Пользователь — просто человек. Он может:
- Разместить задание (нужен креатор)
- Создать профиль (сам является креатором)
- Просматривать задания
- Смотреть каталог креаторов

Один и тот же человек может делать всё перечисленное. Никакого «активного режима», никакого выбора.

### Аналогия

| Платформа | Как работает |
|---|---|
| OLX | Нажал «Разместить» → выбрал категорию → заполнил форму |
| HH.ru | Нажал «Добавить резюме» → заполнил → опубликовал |
| **viral-wall (новое)** | Нажал «Создать задание» → заполнил → оплатил |

---

## 📦 Два типа публикаций (отдельные сущности)

### Тип 1: Задание (`Ad` / Job)

Кто создаёт: **любой пользователь**, кому нужен создатель контента

```
Примеры:
- Кофейня «Ищу TikTok-видео для рекламы пекарни»
- Магазин «Нужны Reels-обзоры одежды»
- Ресторан «Ищу создателя атмосферного контента»
```

**Монетизация:** обязательная платная публикация (990 ₸ / 7 дней)

### Тип 2: Профиль креатора (`CreatorProfile`)

Кто создаёт: **любой пользователь**, кто создаёт контент

```
Примеры:
- «Снимаю TikTok-видео, 200k подписчиков, от 15 000 ₸»
- «Создаю Instagram Reels, категории: еда, lifestyle»
```

**Монетизация:** опциональная платная публикация в каталоге

---

## 🗄️ Изменения в типах данных

### Тип `User` — упрощение

**Было:**
```typescript
interface User {
  isCreator: boolean;
  isAdvertiser: boolean;
  activeRole: 'creator' | 'advertiser'; // ← УБРАТЬ
}
```

**Станет:**
```typescript
interface User {
  id: string;
  name: string;
  email: string;
  phone: string;
  avatar: string | null;
  createdAt: string;
  // Нет activeRole, нет isCreator, нет isAdvertiser
  // Система определяет наличие профилей через relations/отдельные запросы
}
```

Удалить тип `UserRole = 'creator' | 'advertiser'` полностью.

## 🔐 Изменения в регистрации

### Было (3 шага):
```
Данные → Выбор роли → Профиль роли → Старт
```

### Станет (1 шаг):
```
Данные (имя, телефон, email, пароль) → ✅ Аккаунт создан → Главная
```

Никакого выбора роли при регистрации. Страница [`/auth/role`](../app/auth/role/page.tsx) — удаляется.

**После регистрации:** пользователь видит главную с двумя призывами:
- «Разместить задание» → создаёт задание (с онбордингом если нужно)
- «Создать профиль креатора» → создаёт профиль

## 🚀 Ленивый онбординг (Lazy Onboarding)

### При создании задания (первый раз)

Если у пользователя нет бизнес-профиля (контактная информация для объявлений):

```
Шаг 0 (новый, только первый раз):
  «Как к вам обращаться?»
  → Название / Имя: [_______]
  → Telegram для входящих: [@_______]
  
Шаг 1: Основная информация задания
  ...обычный флоу создания объявления...
```

Шаг 0 — **один раз**. Потом при создании следующего задания сразу Шаг 1.

### При создании профиля (первый раз)

Обычный онбординг [`/onboarding/creator`](../app/onboarding/creator/page.tsx) — остаётся как есть, вызывается при нажатии «Создать профиль».

## 🏷️ Терминология в UI

| Старое | Новое | Почему |
|---|---|---|
| «Режим бизнеса» | «Создать задание» | Конкретное действие |
| «Режим креатора» | «Мой профиль» | Конкретный объект |
| «Переключить роль» | (удалено) | Не нужно |
| «Активировать бизнес» | (удалено) | Не нужно |
| «Кабинет бизнеса» | «Мои задания» | Понятнее |
| «Профиль» (в Nav) | «Мой профиль» | Понятнее |
| «Объявления» | «Задания» | Точнее описывает суть |

> **Примечание по терминологии:** «Задание» лучше чем «Объявление» потому что передаёт суть — бизнес ищет исполнителя для конкретной задачи. Но если менять везде — слишком много работы. В MVP можно оставить «Объявление» в коде, но в UI добавить подзаголовок «Задание для креатора».

## 🔄 Схема миграции данных (для бэкенда)

```sql
-- Убрать из таблицы users:
ALTER TABLE users DROP COLUMN active_role;
-- (is_creator и is_advertiser можно оставить как денормализованный кэш,
--  но не использовать для управления доступом)

-- Логика доступа через relations:
-- Может создавать задания? → проверяем EXISTS (SELECT 1 FROM ads WHERE owner_id = user_id)
-- Есть профиль? → проверяем EXISTS (SELECT 1 FROM creator_profiles WHERE user_id = ?)
```

## 📊 Влияние на монетизацию

**Ничего не меняется.** Модель остаётся той же:
- Задание (объявление) — платная публикация 990 ₸ / 7 дней
- Профиль в каталоге — опционально платная публикация

Единственное изменение: **снижается барьер входа**. Пользователю не нужно «настраивать роль» перед тем как заплатить. Прямой путь: пришёл → нажал «Создать задание» → оплатил.

## ✅ Ожидаемый результат

**До:**
```
Регистрация (3 шага) → Выбор роли → Онбординг роли → Переключение режимов → Создание задания
```

**После:**
```
Регистрация (1 шаг) → Нажал «Создать задание» → [первый раз: 1 доп. вопрос] → Создание задания → Оплата
```

**Психологически:** пользователь не думает «кто я на этой платформе?», он думает «что я хочу сделать?»
