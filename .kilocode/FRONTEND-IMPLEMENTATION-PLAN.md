# План создания фронтенда Zharnamarket PARTNER

> ⚠️ **Обновлено 25.03.2026** — Убрана система ролей с переключением. Нет `activeRole`. Один аккаунт — любые действия. Множественные профили-анкеты. Подробнее: [`redesign-roles.md`](redesign-roles.md)

## Цель

Собрать современный SSR-ready фронтенд для MVP-модели **Paid Classifieds**:
- бизнес публикует объявление;
- креатор находит объявление или профиль;
- стороны нажимают `Связаться` и уходят в прямую коммуникацию;
- на первом этапе все страницы статические, без подключения бэкенда.

## 2. Почему именно этот стек

### Почему Next.js
- дает SSR из коробки;
- удобная файловая маршрутизация;
- хорошо подходит для SEO публичных страниц;
- легко разделять public/business/creator/admin route groups;
- удобно позже подключать API routes или server actions.

### Почему Tailwind + Ant Design вместе
- Tailwind быстро собирает layout и адаптив;
- Ant Design закрывает сложные элементы: `Form`, `Select`, `Dropdown`, `Table`, `Drawer`, `Modal`, `DatePicker`, `Tabs`, `Steps`, `Upload`;
- такая связка хорошо подходит под marketplace/classified продукт.

### Почему без тяжелого state management
На первом этапе нет backend flow, нет сложной синхронизации данных, поэтому **Redux** сейчас не нужен.

## 7. Что делать на статическом этапе

На первом этапе все страницы должны быть **настоящими**, но без API и бэкенда.

### Уже должны работать
- dropdown;
- select;
- form validation;
- tabs;
- drawer;
- modal;
- datepicker;
- radio/checkbox/switch;
- дропдаун «+ Создать» в шапке (два действия: Задание / Профиль);
- filters через URL/query params;
- fake success/error notifications.

### Пока не должно быть
- реальной авторизации;
- реальной оплаты;
- реальной отправки форм;
- реальных запросов к серверу;
- реального сохранения данных.

### Вместо этого
- mock data;
- local component state;
- fake handlers;
- UI-only flows;
- SSR/SSG для статичных данных.

## 11. UX-правила, которые стоит заложить сразу

### Для объявлений
- фильтры должны быть сверху и работать без бэка;
- детальная объявления должна иметь заметную кнопку `Связаться`;
- статус объявления должен быть виден сразу;
- featured объявления должны визуально выделяться.

### Для профилей креаторов
- карточка должна сразу показывать имя, город, платформы, ставку, availability;
- verified должен быть заметен;
- контакты — только на детальной или через action block.

### Для навигации и действий
- кнопка «+ Создать» в header — дропдаун с двумя опциями: «Задание» и «Профиль»;
- навигация единая для всех пользователей;
- кабинет `/cabinet` — единый, два блока: «Мои задания» + «Мои профили».

## 13. Что не делать на первом фронтовом этапе

Не делать пока:
- реальный backend;
- настоящую auth;
- настоящую оплату;
- API integration;
- сложный state management;
- server actions с записью в БД;
- CMS;
- i18n, если это тормозит старт;
- темную тему, если это отвлекает.

## 14. Технические рекомендации по SSR

### SSR / SSG подход
- публичные страницы объявлений и профилей — SSR/SSG friendly;
- фильтры — через search params;
- интерактивные формы — client components;
- layouts и списки можно рендерить серверно на моках.

### Практически
- page и layout делать server components по умолчанию;
- формы, select, dropdown, datepicker, tabs — выносить в client components;
- mock data импортировать в server pages и пробрасывать в client UI.

## 15. Итоговая рекомендация

Лучший путь для проекта сейчас:

1. Создать **Next.js App Router** проект.
2. Подключить **Tailwind CSS** и **Ant Design** с SSR-поддержкой.
3. Завести route groups для public / creator / business / admin.
4. Вынести типы из продуктовой документации в `src/types`.
5. Подготовить `mock`-слой.
6. Сделать все страницы статическими, но интерактивными.
7. Только после этого подключать backend.

## 16. Короткий вывод

Для вашего MVP лучше всего подходит **Next.js + TypeScript + Tailwind + Ant Design**.

Это даст:
- SSR;
- современную архитектуру;
- быстрый темп разработки;
- удобные формы и dropdown-компоненты;
- легкий переход от статичного фронта к полноценному продукту.
