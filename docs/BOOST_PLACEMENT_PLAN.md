## 2. Проблемы текущего размещения

| # | Проблема | Влияние |
|---|---|---|
| P1 | Кнопка спрятана в дропдауне (desktop) | CTA в подменю теряют 50–70% кликов vs primary-кнопка |
| P2 | Единственная точка входа — `/creators/manage` | Юзер на edit/preview/cabinet не видит путь к монетизации |
| P3 | При активном бусте кнопка полностью исчезает | Юзер теряет контекст («работает ли?»), нет возможности продлить/апгрейд |
| P4 | Нет CTA в момент пика мотивации (сразу после публикации) | Упущенный peak-end момент — самая высокая конверсия |
| P5 | Нет единой визуальной системы «статус продвижения» | Не выстраивается ментальная модель «где посмотреть мои бусты» |

## 3. Рекомендации (приоритизированы)

### 3.1. Сводная таблица

| Приоритет | Что делать | Где | Ожидаемый эффект |
|---|---|---|---|
| P0 (must) | Вынести кнопку буста из дропдауна в primary CTA на карточке | `/creators/manage` (desktop) | +40–60% кликов по CTA |
| P0 (must) | Добавить экран успеха после публикации с CTA буста | новый: `/creators/[id]/published` или модалка | самая высокая конверсия «тёплого» юзера |
| P1 (should) | Показывать статус активного буста вместо скрытия + CTA «Продлить/Апгрейд» | `/creators/manage` карточка | возвращает возможность апсейла |
| P1 (should) | Sticky-баннер или блок в сайдбаре на странице редактирования | `/creators/edit` | контекстный CTA после улучшения профиля |
| P2 (nice) | Виджет «Продвижение» в главной кабинета | `/cabinet` | повышает discoverability через хаб |
| P2 (nice) | Мини-блок статуса буста на превью | `/creators/[id]/preview` | консистентность информации |
| Never | **НЕ добавлять** CTA на шагах создания 1–4 | `/creators/new` | избегаем pay-wall до ценности |

### 3.2. Почему НЕ показывать во время создания (шаги 1–4)

- **Pay-wall до ценности.** Юзер ещё не увидел результат, платёжный CTA = фрикшен.
- **Completion rate.** Монетизация в онбординге режет завершение на 15–30% (подтверждено в sign-up flow исследованиях Eleken, NN Group).
- **Отвлечение.** У шага уже есть цель (заполнить данные) — любой второй CTA ухудшает focus.

**Правильный момент** — сразу после `publish`, когда юзер увидел свою работу опубликованной.

## 4. Детальный план реализации

### Этап 1. P0 — Manage Page: вынести CTA из дропдауна

**Цель:** сделать кнопку «Продвинуть» видимой первым экраном.

**Изменения:**
1. [app/creators/manage/page.tsx](../app/creators/manage/page.tsx#L292-L302) — убрать пункт из Dropdown, перенести в основное тело карточки.
2. Desktop: добавить primary-кнопку в правом верхнем углу карточки (рядом со статусом).
3. Mobile: кнопка уже есть на строках 620–631 — оставить, но перекрасить в primary.
4. Если буст активен — заменить кнопку на чип/бейдж «Буст: VIP, осталось 5 дн.» с вторичной ссылкой «Продлить».

**Тех. детали:**
- Использовать существующий `RocketOutlined` иконку для консистентности.
- Цвет — фирменный акцент (тот же, что на `/creators/[id]/boost/page.tsx`).
- Кнопка ведёт на существующий маршрут `/creators/[id]/boost` — API не меняется.

**Риски:**
| Риск | Вероятность | Митигация |
|---|---|---|
| Перегруженность карточки на мобилке | средняя | Spacing audit, collapse менее важных actions в «Ещё» |
| Conflict с существующими статусами карточки | низкая | Проверить визуально все состояния: черновик, опубликован, на модерации, активный буст, истёкший буст |
| Юзеры привыкли к дропдауну | низкая | В первом релизе оставить пункт и в дропдауне (дубль) на 2–4 недели, потом удалить |

---

### Этап 2. P0 — Экран успеха после публикации

**Цель:** поймать peak-end момент.

**Варианты реализации:**
- **A.** Новый маршрут `/creators/[id]/published` (после успешного publish редирект сюда вместо `/creators/manage`).
- **B.** Модалка успеха поверх `/creators/manage` после publish.

**Рекомендация:** вариант A — даёт полноэкранный fokus, легче A/B-тестировать.

**Содержание экрана:**
1. Заголовок «Профиль опубликован» + имя креатора.
2. Превью карточки (как её видят другие).
3. Два CTA:
   - **Primary:** «Продвинуть в топ» → `/creators/[id]/boost`
   - **Secondary:** «В кабинет» → `/creators/manage`
4. Optional: краткое объяснение, что даёт буст (1–2 bullets).

**API/Backend:**
- Новых эндпоинтов не требуется (boost endpoint уже есть).
- Логика публикации: [app/api/creators/\[id\]/publish/route.ts](../app/api/creators/[id]/publish/route.ts) — возвращает существующий ответ, фронт делает редирект.

**Риски:**
| Риск | Вероятность | Митигация |
|---|---|---|
| Восприятие как агрессивный апсейл | средняя | Тон формулировок — «помощь», а не «купи». Secondary CTA равновесен visually |
| Юзер ожидает попасть сразу в manage | низкая | Тексты + явная secondary-ссылка «В кабинет» |
| Экран не виден при редактировании уже опубликованного профиля | низкая | Показывать только после **первой** публикации (флаг `first_publish_at` или проверка по времени) |

---

### Этап 3. P1 — Статус активного буста

**Цель:** убрать «дыру» в интерфейсе при активном бусте.

**Изменения:**
1. На карточке `/creators/manage`: показать бейдж `Буст активен • VIP • осталось N дней` (вместо скрытия кнопки).
2. Клик по бейджу → страница деталей буста или `/creators/[id]/boost` с префилом «Продлить».
3. Требуется API для получения текущего активного буста (если ещё нет) — проверить `/api/creators/[id]` или создать endpoint.

**Риски:**
| Риск | Вероятность | Митигация |
|---|---|---|
| Нет поля с текущим бустом в API ответе | высокая | Ревью [app/api/creators/\[id\]/route.ts](../app/api/creators/[id]/route.ts), расширить response |
| Юзер покупает второй буст поверх активного | средняя | На чекауте показать «У вас уже есть активный буст — это продлит срок» |
| Таймзоны в «осталось N дней» | средняя | Использовать UTC на бэке, клиентскую локаль на фронте |

---

### Этап 4. P1 — Sticky-блок на Edit

**Цель:** контекстный CTA в момент улучшения профиля.

**Изменения:**
1. [app/creators/edit/page.tsx](../app/creators/edit/page.tsx) — добавить sticky-сайдбар справа или sticky-баннер сверху.
2. Условие показа: профиль уже опубликован И нет активного буста.
3. Текст: «Обновил профиль? Продвинь его, чтобы больше людей увидели».

**Риски:**
| Риск | Вероятность | Митигация |
|---|---|---|
| Отвлекает от редактирования | средняя | Вариант свёрнутого маленького баннера, раскрывается по клику |
| Показывается черновикам | низкая | Проверка `status === 'published'` обязательна |
| Конфликт с мобильной вёрсткой | средняя | На мобилке — не sticky, а в конце страницы после формы |

---

### Этап 5. P2 — Виджет в кабинете

**Цель:** «хаб» для всех финансовых/продвиженческих действий в одном месте.

**Изменения:**
1. [app/cabinet/page.tsx](../app/cabinet/page.tsx) — новый виджет «Продвижение» рядом с балансом.
2. Содержит: список профилей/объявлений с статусами буста + кнопка «Продвинуть ещё».

**Риски:**
| Риск | Вероятность | Митигация |
|---|---|---|
| Дублирование функциональности с `/creators/manage` | средняя | Виджет — краткий summary + deep-link, а не полное управление |
| Перегруз кабинета | средняя | Ограничить 3 записями, «Смотреть все» ссылкой |

## 5. Общие риски проекта

| Категория | Риск | Митигация |
|---|---|---|
| **Продукт** | Буст воспринимается как «платный спам» | Прозрачность: показывать чем буст помогает, какие есть бесплатные альтернативы |
| **Продукт** | Креаторы без бюджета чувствуют себя ущемлёнными | Free tier виcibility, честный ranking не «только буст» |
| **Технический** | Рассинхрон статуса буста между клиентом и сервером | SWR/revalidation на критичных экранах, webhook от платёжки |
| **Технический** | Двойные списания при слабом соединении | Идемпотентность платёжного endpoint (ключ на транзакцию) |
| **Юридический / KZ** | Публикация цен и условий без оферты | Линк на оферту в чекауте (проверить [app/creators/\[id\]/boost/page.tsx](../app/creators/[id]/boost/page.tsx)) |
| **UX** | Разный UX для буста креатора vs буста объявления | Унифицировать компоненты чекаута (shared component) |
| **Аналитика** | Нельзя будет измерить эффект изменений | Ввести события до релиза: `boost_cta_view`, `boost_cta_click`, `boost_purchase_complete` с указанием placement |

## 6. Метрики успеха

Что мерить после релиза каждого этапа:

1. **CTR на кнопку буста** по каждому placement отдельно.
2. **Конверсия placement → checkout → оплата** (воронка).
3. **Completion rate создания профиля** (убедиться, что не ухудшился).
4. **% профилей с активным бустом** (абсолютный рост).
5. **Retention буста** (процент продливающих).
6. **NPS / user feedback** — нет ли жалоб на «агрессивный апсейл».

## 7. Порядок релизов (предлагаемый)

1. **Спринт 1:** Этап 1 (manage CTA) + аналитика (обязательно).
2. **Спринт 2:** Этап 2 (экран успеха).
3. **Спринт 3:** Этап 3 (статус активного буста).
4. **Спринт 4:** Этапы 4 и 5 (edit-блок + кабинет-виджет).

Между спринтами — 2 недели сбора данных перед следующим изменением, чтобы не размывать атрибуцию.
