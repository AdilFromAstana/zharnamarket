# Plan: Облегчить Step4Payment

---

## UX-анализ: выбор способа оплаты до или после кнопки "Оплатить"?

### Исследование

Источники: Baymard Institute (200 000+ часов UX-тестирования), NN/g (Nielsen Norman Group).

### Контекст: это НЕ обычный e-commerce checkout

Текущая страница — оплата **одной фиксированной услуги** (публикация за 990 ₸). Здесь нет:
- Адреса доставки
- Выбора доставки
- Корзины с товарами

Это ближе к **micro-payment** (оплата подписки, буста, публикации) — как в App Store, Spotify, или Telegram Premium.

### Что говорит исследование

**Baymard Institute** (2023–2025):
- Интерфейс выбора оплаты (radio, tabs) сам по себе **не создаёт проблем** — пользователи справляются с любым вариантом
- Но **11% пользователей бросают** корзину если нет их способа оплаты → значит способы показать **надо**, вопрос — когда
- **Ключевое**: текст кнопки должен меняться при выборе метода (не "Оплатить", а "Оплатить через Kaspi" / "Списать из кошелька")
- Необязательные поля (промокод, "Address Line 2") лучше **прятать за ссылку** — показывать по клику

**NN/g** (Minimize Cognitive Load):
- Каждый визуальный элемент, решение, поле = расход "мозговой мощности"
- Принцип **"Offload tasks"**: если можно установить дефолт — установи; если решение не критично — спрячь
- Принцип **"Avoid visual clutter"**: лишние блоки замедляют пользователя даже если он их не использует

### Три варианта и их оценка

#### Вариант A: Всё на одной странице (как сейчас)
```
[Что оплачивается: 990 ₸]
[Промокод]
[Итого]
[Способ оплаты: Кошелёк / Kaspi / Halyk / Карта]
[Безопасная оплата]
[Кнопка: Оплатить]
```
- (+) Всё видно сразу, нет скрытых шагов
- (-) Страница длинная и тяжёлая (что мы видим на скриншоте)
- (-) 4 карточки оплаты = визуальный шум для простой операции

#### Вариант B: Оплатить → потом выбрать метод (предложение пользователя)
```
[Что оплачивается: 990 ₸]
[Промокод (ссылка)]
[Итого]
[Кнопка: Оплатить 990 ₸]
    ↓ клик
[Модалка/шит: выбор метода]
```
- (+) Чистая страница, фокус на сумме
- (+) Паттерн Stripe Checkout, Apple Pay — юзеры привыкли
- (-) Лишний шаг/клик
- (-) Пользователь не видит заранее, хватает ли денег на кошельке

#### Вариант C (РЕКОМЕНДУЕМЫЙ): Smart Default + прогрессивное раскрытие
```
[Что оплачивается: 990 ₸]
[Есть промокод? (ссылка)]
[Итого: 990 ₸]
─────────────────────────
[Кошелёк: 5 000 ₸ ✓  ]  ← auto-selected если хватает
  Другой способ оплаты ↓  ← ссылка, по клику раскрывает остальные
─────────────────────────
[Кнопка: Списать из кошелька 990 ₸]

ⓘ После оплаты объявление появится в ленте на 7 дней.
```

**Если на кошельке недостаточно:**
```
[Кошелёк: 200 ₸ — не хватает]
  Пополнить на 790 ₸ →
──
○ Kaspi Pay
○ Halyk Bank  
○ Банковская карта
─────────────────────────
[Кнопка: Оплатить 990 ₸ через Kaspi Pay]
```

**Почему этот вариант лучший:**
1. **Smart default** — если на кошельке хватает, сразу выбран кошелёк. Не нужно ничего решать → минимум cognitive load
2. **Прогрессивное раскрытие** — другие методы видны только тем, кому нужны (по клику на "Другой способ")
3. **Баланс виден сразу** — в отличие от вар. B, юзер видит что денег хватает ещё до нажатия
4. **Кнопка адаптивна** — текст меняется: "Списать из кошелька 990 ₸" / "Оплатить через Kaspi Pay 990 ₸" (рекомендация Baymard)
5. **Промокод свёрнут** — не занимает место (рекомендация Baymard: необязательные поля → за ссылку)
6. **Alert заменён на строку текста** — меньше визуального шума

## Проблемы текущей версии (по скриншоту)

1. **Ценовой блок** — большой круг-иконка + цена `text-2xl` = громоздкий
2. **Два `<Divider />`** — жирные Ant Design линии
3. **Промокод** — всегда раскрыт, занимает место
4. **«К оплате 990 ₸»** — дублирует цену
5. **4 карточки оплаты** — стопка прямоугольников `rounded-xl border p-3`
6. **Alert** — тяжёлый Ant Design Alert
7. **Серые иконки** — Tailwind v4 vs Ant Design specificity

## Реализация (Вариант C)

### 1. Компактный ценовой блок
- Иконка: `w-8 h-8` вместо `w-10 h-10`
- Цена: `text-xl` вместо `text-2xl`
- Padding: `p-3 gap-3` вместо `p-4 gap-4`

### 2. Убрать Divider
- Заменить `<Divider />` на `<div className="border-t border-gray-100" />` или убрать — spacing достаточно

### 3. Промокод — свёрнут по умолчанию
- Показать кликабельный текст «Есть промокод?» (стиль ссылки)
- По клику → раскрыть input + кнопку
- Добавить `useState(promoExpanded)` внутри Step4Payment
- Если промокод уже применён (`promoResult?.valid`) — показывать результат всегда

### 4. Способы оплаты — smart default + progressive disclosure
- **Если `walletBalance >= requiredAmount`**:
  - Показать компактную строку: "Кошелёк: {баланс} ₸ ✓" (выбран автоматически)
  - Под ней ссылка: "Другой способ оплаты"
  - По клику ссылки → раскрыть остальные 3 метода (Kaspi, Halyk, карта)
- **Если `walletBalance < requiredAmount`**:
  - Показать "Кошелёк: {баланс} ₸ — не хватает · Пополнить на {разницу} ₸"
  - Ниже сразу показать все 3 альтернативных метода
  - Первый метод (Kaspi) выбран по умолчанию

### 5. Адаптивный текст кнопки (Baymard recommendation)
- Текст кнопки должен отражать выбранный метод:
  - wallet → "Списать из кошелька {сумма}"
  - kaspi → "Оплатить через Kaspi Pay {сумма}"
  - halyk → "Оплатить через Halyk Bank {сумма}"
  - card → "Оплатить картой {сумма}"
- Это делает родительский компонент (передаёт текст в кнопку "Далее")

### 6. Итого — компактнее
- `text-base font-semibold` вместо `text-lg`
- Escrow: `text-xl` вместо `text-2xl`
- Если промокод применён — зачёркнутая старая цена рядом

### 7. Alert → строка подсказки
- Убрать `<Alert>` от Ant Design
- Заменить на:
  ```tsx
  <p className="text-xs text-gray-400 flex items-start gap-1.5 mt-2">
    <SafetyOutlined style={{ color: '#9ca3af' }} className="mt-0.5 shrink-0" />
    После оплаты объявление сразу появится в ленте на 7 дней.
  </p>
  ```

### 8. Иконки — inline style
- Все `@ant-design/icons`: Tailwind цветовые классы → `style={{ color: '...' }}`

### 9. Общие отступы
- `space-y-3` вместо `space-y-4`
- `md:p-5` вместо `md:p-6`
