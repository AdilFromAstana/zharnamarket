# Система Рейтингов/Отзывов для Zharnamarket PARTNER

## Исследование: Кому, куда и зачем нужны отзывы

---

## 1. Контекст проекта

Zharnamarket PARTNER — маркетплейс платных объявлений, соединяющий бизнесы с контент-креаторами в Казахстане.

**Реальная архитектура сущностей:**
| Сущность | Описание | Роль в системе |
|----------|----------|----------------|
| **User** | Базовый аккаунт (роли: `user` / `admin`) | Центральная сущность. Может одновременно быть и заказчиком, и креатором |
| **Ad (Task)** | Объявление/задание (990₸/7 дней) | Принадлежит User напрямую (`ownerId`) |
| **CreatorProfile** | Портфолио креатора (N штук на User) | Самостоятельная публичная сущность с каталогом |
| **AdvertiserProfile** | Доп. инфо о бизнесе (1:1 с User) | **Вспомогательная "визитка"**, НЕ самостоятельная сущность. Хранит companyName, city, contacts |

**Ключевой момент:** AdvertiserProfile — это не отдельный "профиль рекламодателя" типа CreatorProfile. Это лишь дополнительные поля юзера (компания, город, контакты). Объявления привязаны к User, а не к AdvertiserProfile. Публичная страница заказчика `/customers/[id]` строится из User + AdvertiserProfile (если есть).

**Текущие сигналы доверия:** только ручной флаг `verified` от админа — **не масштабируется**.

**Коммуникация:** происходит вне платформы (Telegram/WhatsApp) — платформа не знает, состоялось ли сотрудничество.

## 2. Анализ: Куда нужны отзывы?

### 2.1. CreatorProfile — **ОБЯЗАТЕЛЬНО (MVP)**

**С точки зрения бизнеса:**
- Это **главный барьер доверия**: бизнес не знает, доставит ли креатор качественный контент
- Портфолио (видео, статистика) показывает "что умеет", а отзывы — "каково с ним работать"
- Креаторы с отзывами получают больше заказов → готовы платить за публикацию профиля → **рост монетизации**
- Накопленные отзывы = **lock-in** (креатор с 50 отзывами не уйдёт к конкуренту)

**С точки зрения UX:**
- Бизнес видит реальный опыт других заказчиков → быстрее принимает решение
- Креатор мотивирован работать качественно → улучшает экосистему
- Рейтинг на карточке в каталоге → мгновенный сигнал качества

**Кто оставляет:** бизнесы/заказчики, которые связались с креатором через платформу

**Аналоги:** Fiverr (рейтинг продавца), Kwork, OLX (рейтинг продавца)

### 2.2. User как заказчик (страница `/customers/[id]`) — **POST-MVP (фаза 2)**

> **Важно:** в системе нет отдельного "Advertiser Profile" как самостоятельной сущности. Отзывы на заказчика привязываются к **User.id**, а отображаются на странице `/customers/[id]`.

**С точки зрения бизнеса:**
- Защищает креаторов от мошенников и недобросовестных заказчиков
- Замыкает "петлю доверия" — обе стороны мотивированы вести себя честно
- Заказчики с хорошим рейтингом привлекают лучших креаторов

**С точки зрения UX:**
- Креатор видит рейтинг заказчика перед тем, как связаться → снижает риск
- На карточке объявления отображается рейтинг автора → прозрачность

**Почему post-MVP:**
- Основной поток доверия идёт в сторону "бизнес → креатор" (бизнес выбирает исполнителя)
- Креатор выбирает задание по описанию и бюджету — рейтинг заказчика вторичен
- Усложняет MVP без пропорционального роста ценности
- Можно добавить позже без рефакторинга (через `targetType` enum)

### 2.3. Объявления (Ads/Tasks) — **НЕТ**

**Почему НЕТ:**
- Объявления **временные** — живут 7 дней, потом истекают
- Отзыв на конкретное объявление бессмысленен после его закрытия
- Это задание, а не продавец — отзыв должен идти на автора
- Функция жалоб (`Report`) уже покрывает кейс "плохое объявление"

**Что делать вместо:** отображать рейтинг заказчика (User) **на карточке объявления** (наследуемый рейтинг) — в фазе 2.

### 2.4. User (единый рейтинг) — **НЕТ**

**Почему НЕТ:**
- Один User может быть и креатором, и заказчиком одновременно
- Единый рейтинг смешивает контексты: "4.5 звезды — как креатор или как заказчик?"

**Что делать вместо:** раздельные рейтинги — `creator_profile` (на CreatorProfile.id) и `customer` (на User.id в роли заказчика).

## 3. Итоговая рекомендация

| Сущность | Отзывы? | Фаза | Обоснование |
|----------|---------|------|-------------|
| **CreatorProfile** | **ДА** | **MVP (фаза 1)** | Главный драйвер доверия. Бизнес выбирает креатора — ему нужны отзывы |
| **User-как-заказчик** | **ДА** | **Фаза 2** | Замыкает петлю доверия. targetType = `customer`, targetId = User.id |
| **Ad (Task)** | **НЕТ** | — | Показывать рейтинг автора на карточке (наследуемый) |
| **User (единый)** | **НЕТ** | — | Раздельные рейтинги по контексту |

**MVP-концепция: односторонние отзывы бизнес → креатор**
- Бизнес связывается с креатором → через 3-5 дней может оставить отзыв на CreatorProfile
- Креатор может ответить на отзыв (1 ответ)
- Схема данных заложена под расширение (targetType enum)

## 4. Главная проблема: верификация сотрудничества

Поскольку общение происходит вне платформы (Telegram/WhatsApp), платформа **не знает**, состоялось ли реальное сотрудничество. Нужен механизм валидации.

### Варианты:

| Вариант | Описание | Плюсы | Минусы | Для MVP? |
|---------|----------|-------|--------|----------|
| **A. Contact-Click** | Клик на "Связаться" → через 5-7 дней предложение оставить отзыв | Нулевой friction, использует существующий трекинг | Нет гарантии реального сотрудничества | **ДА** |
| **B. Взаимное подтверждение** | Обе стороны подтверждают сотрудничество → открываются отзывы | Высокое качество отзывов | Высокий friction, многие не подтвердят | Нет |
| **C. Открытые отзывы** | Любой может оставить отзыв | Максимальный охват | Крайне подвержено абузу | Нет |
| **D. Гибрид (A+B)** | Contact-click → обычный отзыв; взаимное подтверждение → "Verified Review" | Лучшее из обоих | Сложнее в реализации | Post-MVP |

### Рекомендация: **Вариант A для MVP**, эволюция в **D** позже

**Механизм:**
1. Пользователь кликает "Связаться" (Telegram/WhatsApp) на объявлении или профиле креатора
2. Записывается `ContactInteraction` (уже частично есть как `contactClickCount`)
3. Через 5-7 дней → уведомление: "Вы связывались с [Имя]. Работали вместе? Оставьте отзыв!"
4. Рецензируемая сторона может ответить на отзыв (1 ответ)

## 5. Модель данных для отзывов

### 5.1. Prisma Schema (MVP — отзывы на CreatorProfile)

```prisma
// Enum с заделом на будущее (фаза 2 добавит "customer")
enum ReviewTargetType {
  creator_profile    // MVP: отзыв на профиль креатора
  // customer        // Фаза 2: отзыв на заказчика (User.id)
}

enum ReviewStatus {
  published
  hidden      // скрыт модерацией
  reported    // на проверке после жалобы
}

model Review {
  id              String            @id @default(cuid())
  
  // Кто оставил отзыв
  reviewerId      String
  reviewer        User              @relation("ReviewsGiven", fields: [reviewerId], references: [id])
  
  // На кого — для MVP всегда creator_profile
  targetType      ReviewTargetType
  targetId        String            // CreatorProfile.id (MVP) | User.id (фаза 2, для customer)
  
  // Прямая связь с CreatorProfile (для MVP)
  creatorProfileId String?
  creatorProfile   CreatorProfile?  @relation(fields: [creatorProfileId], references: [id])
  
  // Рейтинг
  overallRating   Int               // 1-5
  comment         String            // 50-1000 символов
  
  // Ответ от креатора
  response        String?           // текст ответа
  respondedAt     DateTime?
  
  // Мета
  status          ReviewStatus      @default(published)
  createdAt       DateTime          @default(now())
  updatedAt       DateTime          @updatedAt
  
  // Связь с взаимодействием (contact-click)
  interactionId   String?
  interaction     ContactInteraction? @relation(fields: [interactionId], references: [id])
  
  @@unique([reviewerId, targetType, targetId]) // 1 отзыв на пару reviewer + target
  @@index([targetType, targetId])
  @@index([reviewerId])
  @@index([creatorProfileId])
  
  @@map("reviews")
}

// Персональный трекинг contact-click (расширение существующего contactClickCount)
model ContactInteraction {
  id              String    @id @default(cuid())
  
  // Кто нажал "Связаться"
  userId          String
  user            User      @relation(fields: [userId], references: [id])
  
  // С кем связывались
  creatorProfileId String?           // если контакт с креатором
  creatorProfile   CreatorProfile?   @relation(fields: [creatorProfileId], references: [id])
  adId             String?           // если контакт с автором объявления
  ad               Ad?               @relation(fields: [adId], references: [id])
  
  // Владелец контакта (User.id креатора или автора объявления)
  contactedUserId String
  
  contactMethod   String             // "telegram" | "whatsapp" | "phone"
  createdAt       DateTime           @default(now())
  
  // Право на отзыв (через 3 дня после контакта)
  reviewEligibleAt DateTime
  
  reviews         Review[]
  
  @@unique([userId, creatorProfileId])  // 1 interaction на пару user + creator
  @@index([userId])
  @@index([creatorProfileId])
  
  @@map("contact_interactions")
}
```

## 6. Критерии оценки (субрейтинги)

### Для CreatorProfile (заказчик оценивает креатора):
| Критерий | Описание |
|----------|----------|
| **Качество контента** | Соответствие ТЗ, креативность, техническое исполнение |
| **Коммуникация** | Отзывчивость, вежливость, доступность |
| **Соблюдение сроков** | Выполнение в оговорённые сроки |

### Для User-как-заказчика (фаза 2, креатор оценивает заказчика):
| Критерий | Описание |
|----------|----------|
| **Скорость оплаты** | Своевременность и полнота оплаты |
| **Чёткость ТЗ** | Понятность задания, адекватность требований |
| **Коммуникация** | Отзывчивость, конструктивная обратная связь |

**MVP:** только `overallRating` (1-5) + `comment` — без субрейтингов
**Фаза 3:** добавить субрейтинги как JSON поле `criteriaRatings`

## 7. Бизнес-ценность системы отзывов

### 7.1. Прямая ценность

| Метрика | Влияние | Механизм |
|---------|---------|----------|
| **Конверсия в контакт** | +20-40% | Доверие через социальное доказательство |
| **Retention (удержание)** | +30-50% | Lock-in: накопленные отзывы нельзя перенести |
| **LTV пользователя** | +15-25% | Больше повторных публикаций и бустов |
| **Качество сервиса** | Рост | Самоочищение: плохие участники уходят |

### 7.2. Монетизационный потенциал

1. **"Закреплённый отзыв"** — креатор может закрепить лучший отзыв вверху (часть премиум-буста)
2. **"Аналитика отзывов"** — детальная статистика по рейтингам (платная функция)
3. **Приоритет в каталоге** — профили с высоким рейтингом ранжируются выше
4. **"Verified Review" бейдж** — платная верификация сотрудничества

### 7.3. Конкурентное преимущество

- **OLX.kz** — базовые звёзды, не специализированные
- **Zharnamarket** — отраслевые критерии (качество контента, сроки, оплата)
- **Ни один конкурент** в Казахстане не имеет специализированных отзывов для инфлюенсер-маркетинга

### 7.4. Сетевой эффект

```
Больше отзывов → Больше доверия → Больше сделок → Больше отзывов
                                                          ↑
                                          Больше пользователей ←
```

### 8.2. Формы отзыва

**Оставить отзыв (минимальная форма):**
```
┌────────────────────────────────────────┐
│  Оцените сотрудничество с [Имя]       │
│                                        │
│  Общая оценка: ★ ★ ★ ★ ☆  (4/5)      │
│                                        │
│  Ваш отзыв:                           │
│  ┌──────────────────────────────────┐  │
│  │ Отличный креатор! Сделал видео   │  │
│  │ в срок, качество на высоте...    │  │
│  └──────────────────────────────────┘  │
│  Мин. 50 символов                      │
│                                        │
│  [Опубликовать отзыв]                  │
└────────────────────────────────────────┘
```

### 8.3. Защита от абуза (MVP)

| Механизм | Описание | Фаза |
|----------|----------|------|
| **Contact-click prerequisite** | Только тот, кто нажал "Связаться", может оставить отзыв | MVP |
| **Таймер 3 дня** | Нельзя оставить отзыв сразу — нужно время на реальную работу | MVP |
| **1 отзыв на пару** | Один отзыв на пару (reviewer → creatorProfile) | MVP |
| **Мин. длина комментария** | Минимум 50 символов — предотвращает "пустые" отзывы | MVP |
| **Жалоба на отзыв** | Кнопка "Пожаловаться" → админ рассматривает | MVP |
| **Мин. возраст аккаунта** | Аккаунт должен быть старше 3 дней | MVP |
| **Модерация** | Ручная модерация через admin panel | Фаза 2 |

### 8.4. Cold Start (проблема пустых отзывов)

| Решение | Описание |
|---------|----------|
| **"Новый профиль" бейдж** | Вместо пустых звёзд — яркий бейдж "Новичок" |
| **Промо-период** | Первые 30 дней — скидка на публикацию для тех, кто оставляет отзывы |
| **Email-напоминания** | После contact-click → цепочка: 5 дней, 10 дней, 14 дней |
| **Упрощённая форма** | Для первых отзывов — только звёзды, без обязательного текста |

## 9. API Endpoints (план)

### MVP: Reviews API

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/reviews?creatorProfileId={id}` | No | Публичный список отзывов на креатора |
| POST | `/api/reviews` | Yes | Создать отзыв (проверка eligibility через ContactInteraction) |
| PUT | `/api/reviews/[id]/response` | Yes (creator owner) | Креатор отвечает на отзыв |
| POST | `/api/reviews/[id]/report` | Yes | Пожаловаться на отзыв |
| GET | `/api/reviews/my` | Yes | Мои отзывы (оставленные + полученные) |
| GET | `/api/reviews/eligibility?creatorProfileId={id}` | Yes | Могу ли я оставить отзыв? (есть ли ContactInteraction + прошло ли 3 дня) |

### MVP: Contact Interactions (расширение существующего contact-click)

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/creators/[id]/contact-click` | Yes* | Записать ContactInteraction с креатором |

> *Изменение: текущий `/api/tasks/[id]/contact-click` анонимный и считает только число. Для креаторов нужен auth для персонального трекинга. Существующий endpoint для ads можно оставить как есть (без изменений для MVP).

### Фаза 2: Admin Reviews API

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/admin/reviews` | Admin | Все отзывы с фильтрами |
| PATCH | `/api/admin/reviews/[id]` | Admin | Изменить статус (hide/publish) |

## 10. Фазы реализации

### Фаза 1: MVP — отзывы на креаторов (1.5-2 недели)

**Scope:**
- Prisma: модели `Review` + `ContactInteraction`, поля `averageRating`/`reviewCount` в CreatorProfile
- API: `/api/creators/[id]/contact-click` (персональный трекинг)
- API: `/api/reviews` (CRUD + eligibility)
- UI: секция "Отзывы" на странице `/creators/[id]`
- UI: звёзды + count на `CreatorCard` в каталоге `/creators`
- UI: форма создания отзыва (модалка или отдельная секция)
- UI: ответ креатора на отзыв
- UI: "Мои отзывы" в кабинете `/cabinet`
- Валидация: contact-click + 3 дня + 1 отзыв на пару + мин. 50 символов

**Ограничения MVP:**
- Только отзывы на CreatorProfile (не на заказчиков)
- Только overallRating (без субрейтингов)
- Без email-уведомлений
- Без admin-модерации отзывов (только кнопка "Пожаловаться" → Report)

### Фаза 2: Отзывы на заказчиков + модерация (1-2 недели)

- Добавить `customer` в enum `ReviewTargetType`
- Креаторы могут оставить отзыв на User (заказчика) после contact-click на его объявление
- Рейтинг заказчика на карточках объявлений (`AdCard`, `AdDetailClient`)
- Секция отзывов на странице `/customers/[id]`
- Admin panel: модерация отзывов (`/api/admin/reviews`)

### Фаза 3: Продвинутые функции (2-3 недели)

- Субрейтинги (критерии: качество, коммуникация, сроки)

## 12. Резюме

**MVP: Отзывы на CreatorProfile**
- Заказчик связывается с креатором → через 3 дня может оставить отзыв (1-5 звёзд + комментарий)
- Креатор может ответить на отзыв
- Рейтинг отображается на карточках и странице креатора
- Верификация через contact-click (персональный трекинг)

**Фаза 2: Отзывы на заказчиков (User)**
- targetType = `customer`, targetId = User.id
- Отображение на `/customers/[id]` и карточках объявлений

**Отзывы НЕ нужны на:**
1. ~~Объявления (Ad)~~ — временные, 7 дней. Показывать рейтинг автора
2. ~~User (единый рейтинг)~~ — смешивает контексты. Раздельные рейтинги по ролям

**Ключевые решения:**
- Contact-click = триггер права на отзыв (решает проблему верификации без внутренних транзакций)
- `targetType` enum = расширяемая схема без рефакторинга
- Агрегированные поля на CreatorProfile = быстрые запросы каталога без JOIN

**Бизнес-эффект:** retention через lock-in, конверсия через доверие, монетизация через premium features, конкурентное преимущество в инфлюенсер-маркетинге КЗ.
