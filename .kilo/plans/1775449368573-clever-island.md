## 1. ПРОДУКТ

### Назначение
**Zharnamarket PARTNER** — маркетплейс-платформа для связи бизнесов (рекламодателей) с контент-креаторами (блогерами/инфлюенсерами) в Казахстане. Домен: `viraladds.kz`.

### Тип
**Веб-приложение** (адаптивный дизайн под мобильные устройства). Нативных мобильных приложений нет. Весь UI на русском языке (рынок Казахстана), валюта — KZT (тенге).

### Основной workflow

**Для рекламодателя (Advertiser):**
1. Регистрация → `POST /api/auth/register` (email, телефон, пароль) или Google OAuth
2. Онбординг бизнеса → `/onboarding/business` (создание AdvertiserProfile: компания, город, контакты)
3. Создание объявления → `/ads/new` (5-шаговый визард: онбординг → основные данные → детали → превью → оплата)
4. Оплата публикации → 990₸ за 7 дней (Kaspi Pay / Halyk Bank / карта)
5. Управление объявлениями → `/ads/manage` (пауза, возобновление, архивирование, удаление)
6. Продвижение (буст) → `/ads/[id]/boost` (3 тарифа: rise 1990₸, highlight 2990₸, premium 3990₸)

**Для креатора (Creator):**
1. Регистрация → аналогично рекламодателю
2. Создание профиля креатора → `/creators/new` (данные, платформы, портфолио, прайс-лист)
3. Публикация профиля в каталоге
4. Ожидание контакта от рекламодателей

**Общий поток:**
- Рекламодатель ищет креаторов в каталоге `/creators` с фильтрами (город, платформа, категория, бюджет)
- Креатор ищет задания в каталоге `/ads` с фильтрами
- Связь через контактные данные (Telegram, WhatsApp, телефон, email) — **без встроенного мессенджера**

## 2. РОЛИ И ПОЛЬЗОВАТЕЛИ

### Концепция ролей

**User (Пользователь)** — это человек на платформе. Один аккаунт = один человек. Внутри аккаунта пользователь совмещает **две функциональных роли** одновременно:

```
User (человек на платформе)
├── Роль "Работодатель" → публикует задания (Ad[])
│   └── У юзера есть массив объявлений (ads)
├── Роль "Креатор" → публикует профили-резюме (CreatorProfile[])
│   └── У юзера есть массив профилей (creatorProfiles) со своими навыками, портфолио, достижениями
└── Роль "Админ" → модерация, управление промокодами и контентом
    └── НЕ РЕАЛИЗОВАНА в коде (нужно добавить)
```

### Роли в системе

| Роль | Суть | Данные в БД | Статус |
|------|------|-------------|--------|
| **User (Пользователь)** | Человек на платформе. Базовый аккаунт с email, телефоном, паролем | Модель `User` | Реализован |
| **Работодатель (Advertiser)** | Функциональная роль юзера. Публикует задания для креаторов | `User.ads[]` (массив объявлений) + `AdvertiserProfile` (профиль компании — опционально) | Реализован |
| **Креатор (Creator)** | Функциональная роль юзера. Публикует профили-резюме со своими навыками, платформами, портфолио и прайсом | `User.creatorProfiles[]` (массив профилей креатора) | Реализован |
| **Админ (Admin)** | Модерация контента, управление промокодами, разрешение споров | В User **нет поля** `role`/`isAdmin`. Эндпоинты `/api/admin/*` существуют, но проверки роли нет — `// TODO: добавить проверку admin-роли` | **Не реализован** |

> **Ключевой принцип:** Работодатель и Креатор — это не отдельные типы аккаунтов, а две стороны одного пользователя. Любой юзер может одновременно иметь массив заданий (как работодатель) и массив профилей-резюме (как креатор). Переключения между ролями нет — обе доступны из одного кабинета.

### Регистрация и верификация

**Регистрация** (`POST /api/auth/register`):
- Поля: имя, email (unique), телефон (+7XXXXXXXXXX, unique), пароль (мин. 8 символов)
- Пароль хешируется bcrypt с cost factor 12
- Rate limit: 5 регистраций за 60 минут на IP
- Альтернатива: Google OAuth (`GET /api/auth/google`)

**Верификация:**
- Email-верификации **НЕТ** — пользователь сразу активен после регистрации
- Поле `verified` есть в `AdvertiserProfile` и `CreatorProfile`, но механизм верификации не реализован (устанавливается только через seed/вручную)

### Разграничение прав доступа

| Уровень | Механизм | Файл |
|---------|----------|------|
| API-маршруты | JWT Bearer token → `getCurrentUserId()` | `lib/auth.ts` |
| Владение ресурсом | Проверка `ownerId === userId` в каждом роуте | Все CRUD-роуты |
| Клиентская навигация | `useRequireAuth()` хук → редирект на `/auth/login` | `hooks/useRequireAuth.ts` |
| Серверный прокси | Cookie `vw_auth_flag` (легко обходится) | `proxy.ts` |

## 3. МОНЕТИЗАЦИЯ

### Платёжные системы

Определены 3 метода оплаты (`PaymentMethod` enum):
| Метод | Описание |
|-------|----------|
| `kaspi` | Kaspi Pay |
| `halyk` | Halyk Bank |
| `card` | Visa/Mastercard/MИР |

> **КРИТИЧНО:** Реальная интеграция с платёжными провайдерами **НЕ реализована**. Все платежи автоматически помечаются как `"success"`. В коде присутствуют TODO-комментарии:
> - `app/api/payments/ads/[id]/publish/route.ts:117` — `// TODO: здесь интеграция с реальным платёжным провайдером (Kaspi/Halyk)`
> - `app/api/payments/ads/[id]/boost/route.ts:130` — `// TODO: интеграция с платёжным провайдером`

### Бизнес-модель

| Услуга | Цена | Срок | Тип |
|--------|------|------|-----|
| Публикация объявления | 990 ₸ | 7 дней | `ad_publication` |
| Буст "Поднять" (rise) | 1 990 ₸ | 7 дней | `ad_boost` |
| Буст "Выделить" (highlight) | 2 990 ₸ | 7 дней | `ad_boost` |
| Буст "Премиум" (premium) | 3 990 ₸ | 7 дней | `ad_boost` |
| Публикация профиля креатора | Определён `PaymentType.creator_publication`, но **эндпоинт не реализован** | — | — |

### Промокоды
- Типы скидок: `percent` (процент) и `fixed_amount` (фиксированная сумма в тенге)
- Применяются атомарно в транзакции (защита от race condition)
- Если скидка 100% — платёжный провайдер пропускается
- Admin CRUD: `/api/admin/promo` (но без проверки роли!)

### Денежный поток
Платформа работает по модели **комиссии за размещение** — пользователь платит за публикацию и продвижение объявлений. Прямых переводов между пользователями **нет**. Встроенного escrow/safe-deal нет.

export async function signAccessToken(userId: string, email: string, role: string) {
  return new jose.SignJWT({ sub: userId, email, role })
    .setProtectedHeader({ alg: "HS256" })
    .setIssuedAt()

    .sign(SECRET);

Добавить хелпер `isAdmin()`:

```ts
export async function isAdmin(req: Request): Promise<boolean> {
  const token = extractBearerToken(req);
  if (!token) return false;
  try {
    const { payload } = await jose.jwtVerify(token, SECRET);
    return payload.role === "admin";
  } catch {
    return false;
  }
}

// Хелпер для API-роутов — выбрасывает 403 если не админ
export async function requireAdmin(req: Request): Promise<string> {
  const userId = await getCurrentUserId(req);
  if (!userId) throw new Error("UNAUTHORIZED");
  const token = extractBearerToken(req)!;
  const { payload } = await jose.jwtVerify(token, SECRET);
  if (payload.role !== "admin") throw new Error("FORBIDDEN");
  return userId;
}
```

### ФАЗА 5: Бесплатная публикация и скидки

#### Как это работает (уже частично реализовано!)

Механизм бесплатной публикации **уже заложен в коде** через промокоды:

1. Админ создаёт промокод: `{ code: "FREE100", discountType: "percent", discountValue: 100, applicableTo: ["ad_publication"] }`
2. Юзер вводит промокод при оплате публикации
3. `validatePromoCode()` вычисляет: `finalAmount = 0`
4. Платёжный роут видит `finalAmount === 0` → пропускает платёжный провайдер → сразу активирует объявление
5. Запись `PaymentSession` создаётся с `amount: 0, originalAmount: 990, discountAmount: 990`

**Что нужно доделать:**

| Задача | Статус | Детали |
|--------|--------|--------|
| Промокод на публикацию объявления (100%) | Работает | `applicableTo: ["ad_publication"]`, `discountValue: 100` |
| Промокод на буст (100%) | Работает | `applicableTo: ["ad_boost"]`, `discountValue: 100` |
| Промокод на профиль креатора | **Не работает** | `PaymentType.creator_publication` определён, но API-эндпоинт `/api/payments/creators/[id]/publish` не существует |
| Скидка N% на публикацию | Работает | `discountType: "percent"`, `discountValue: N` |
| Скидка N₸ на публикацию | Работает | `discountType: "fixed_amount"`, `discountValue: N` |

**Для полной поддержки нужно (вне текущего скоупа, но описано):**
- Создать `POST /api/payments/creators/[id]/publish` — аналог `ads/[id]/publish` но для профиля креатора
- Добавить шаг оплаты в форму создания профиля креатора (`app/creators/new/page.tsx`)

#### 1.4 Создать `tests/helpers/auth.ts` — хелперы для тестов

**Новый файл:** `tests/helpers/auth.ts`

Функции:
- `createTestUser(overrides?)` — создаёт юзера в БД, возвращает `{ user, accessToken, refreshToken }`
- `createAdminUser(overrides?)` — создаёт admin-юзера, возвращает токены
- `makeRequest(handler, method, url, options?)` — оборачивает вызов route handler в `NextRequest`, возвращает parsed response
- `cleanupTestUsers(emails[])` — удаляет тестовых юзеров по email из БД после тестов

```ts
import { prisma } from "@/lib/prisma";
import bcrypt from "bcryptjs";
import { signAccessToken, signRefreshToken } from "@/lib/auth";
import { NextRequest } from "next/server";

export async function createTestUser(overrides = {}) {
  const email = `test-${Date.now()}-${Math.random().toString(36).slice(2)}@test.local`;
  const password = await bcrypt.hash("TestPass123", 12);
  const user = await prisma.user.create({
    data: { email, name: "Test User", password, role: "user", ...overrides },
  });
  const accessToken = await signAccessToken({ sub: user.id, email: user.email, role: user.role });
  const refreshToken = await signRefreshToken({ sub: user.id, email: user.email, role: user.role });
  return { user, accessToken, refreshToken };
}

export async function createAdminUser(overrides = {}) {
  return createTestUser({ role: "admin", name: "Test Admin", ...overrides });
}

export function buildRequest(method: string, url: string, options?: {
  body?: unknown;
  token?: string;
  headers?: Record<string, string>;
}): NextRequest {
  const init: RequestInit = { method, headers: {} };
  if (options?.body) {
    init.body = JSON.stringify(options.body);
    (init.headers as Record<string, string>)["content-type"] = "application/json";
  }
  if (options?.token) {
    (init.headers as Record<string, string>)["authorization"] = `Bearer ${options.token}`;
  }
  return new NextRequest(new URL(url, "http://localhost:3000"), init);
}

export async function cleanup(userIds: string[]) {
  await prisma.user.deleteMany({ where: { id: { in: userIds } } });
}
```
