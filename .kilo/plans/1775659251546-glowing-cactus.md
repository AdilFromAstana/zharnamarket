# avatarColor в БД: хранить цвет градиента аватара на уровне пользователя

## Концепция

При создании пользователя в системе случайно выбирается 1 из 5 градиентов и записывается в `User.avatarColor`.
При рендере аватара:
- Есть фото → показываем фото (круглое)
- Нет фото → показываем градиент из `avatarColor` с инициалом

Цвет стабильный, одинаковый везде (каталог, детальная страница, хедер, отзывы, профиль).

## Фаза 1: Prisma schema + миграция

### 1.1 Добавить поле в User

**Файл:** `prisma/schema.prisma` (модель User, после строки `avatar`)

```prisma
avatarColor String?
```

Nullable — чтобы миграция не требовала дефолт для существующих записей.

### 1.2 Миграция + бэкфил

```bash
npx prisma migrate dev --name add-avatar-color
```

После создания миграции — добавить SQL бэкфила существующих пользователей прямо в миграционный файл:

```sql
-- Бэкфил: присваиваем детерминированный цвет по хэшу ID
UPDATE users SET "avatarColor" = CASE (ascii(substring(id from 1 for 1)) % 5)
  WHEN 0 THEN 'from-sky-400 to-blue-500'
  WHEN 1 THEN 'from-violet-400 to-purple-500'
  WHEN 2 THEN 'from-emerald-400 to-teal-500'
  WHEN 3 THEN 'from-rose-400 to-pink-500'
  WHEN 4 THEN 'from-amber-400 to-orange-500'
END
WHERE "avatarColor" IS NULL;
```

## Фаза 2: Установка цвета при создании пользователя

### 2.1 Общая утилита

**Файл:** `lib/utils.ts` — добавить (рядом с `getAvatarGradient`):

```ts
export const AVATAR_GRADIENTS = [
  "from-sky-400 to-blue-500",
  "from-violet-400 to-purple-500",
  "from-emerald-400 to-teal-500",
  "from-rose-400 to-pink-500",
  "from-amber-400 to-orange-500",
];

export function randomAvatarGradient(): string {
  return AVATAR_GRADIENTS[Math.floor(Math.random() * AVATAR_GRADIENTS.length)];
}
```

# (Предыдущий план) Обзор и сравнение фильтров: Креаторы vs Объявления

## Сводная таблица отличий

| Функция | Объявления (Ads) | Креаторы (Creators) | Проблема |
|---------|-----------------|---------------------|----------|
| **Мультивыбор** | Можно выбрать несколько значений (массивы `string[]`) | Только одно значение (single `string`) | Нельзя выбрать TikTok + Instagram одновременно |
| **Кол-во (faceted counts)** | Рядом с каждым фильтром показывается число (`TikTok 42`) | Нет кол-ва совсем | Пользователь не видит сколько результатов у каждого фильтра |
| **Общее кол-во** | Серверный `pagination.total` с правильным склонением (`7 942 объявления`) | `creators.length` — длина клиентского массива (макс 50) | Показывает максимум 50 вместо реального числа |
| **URL синхронизация** | Фильтры сохраняются в URL (`/ads?city=Almaty&platform=TikTok`) | Фильтры НЕ сохраняются в URL | Нельзя поделиться ссылкой с фильтрами, при обновлении страницы фильтры сбрасываются |
| **Пагинация** | Полная пагинация (`SEOPagination`, по 20 штук) | Нет пагинации, грузит все до limit=50 | При >50 креаторов часть не показывается |
| **SSR с фильтрами** | Парсит фильтры из URL, SSR рендерит с фильтрами | SSR всегда грузит первые 20 без фильтров | SEO не индексирует отфильтрованные страницы |
| **Disabled состояние** | Фильтры с count=0 серые и некликабельные | Нет disabled состояния | Можно выбрать фильтр и получить 0 результатов |
| **Динамические фильтры** | `DynamicFacetSection` (видеоформат, формат рекламы, предмет) | Все фильтры статические | — |
| **Badge (счетчик)** | Считает все отдельные значения (`countSelections()`) | Считает только наличие ключа (`Boolean`) | Выбрав 3 города, badge покажет 1 вместо 3 |
| **Router Cache sync** | `isSyncingFromServer` ref для корректного поведения при навигации | Простой `isFirstRender`, нет обработки Router Cache | Возможны баги при навигации назад/вперед |

### Шаг 2: API — `app/api/creators/route.ts`

**Что делаем:**
- Поддержка comma-separated массивов в query params (`?city=Almaty,Astana`)
- Prisma `where` clause: `{ in: [...] }` вместо прямого сравнения
- Добавить faceted counts через `groupBy` (4 dimensions: platform, city, category, availability)
- Ответ: `{ data, pagination, facets }`

**Паттерн facets (из ads API `app/api/tasks/route.ts:124-242`):**
Каждый facet использует ВСЕ фильтры КРОМЕ своего dimension — стандартный faceted search.

Для креаторов: `platform` считаем через `platforms` relation (M2M), поэтому вместо `groupBy` нужен отдельный подход — raw query или подсчет через `findMany` + группировку. Остальные 3 измерения (city, category, availability) — обычные `groupBy`.

**Особенность platform:**
У creators `platform` — это M2M relation (`platforms: { some: { name } }`), не прямое поле. `groupBy` по нему невозможен. Решение: отдельный запрос — `prisma.creatorPlatform.groupBy({ by: ["name"], where: { creator: facetWhere } })`.

**Особенность category:**
`contentCategories` — это `String[]` (PostgreSQL array). `groupBy` по нему невозможен. Решение: raw SQL запрос `SELECT unnest("contentCategories") as cat, count(*) FROM ... GROUP BY cat`.
