# SEO-пагинация для страницы /ads

## Контекст

Текущее состояние:
- `page.tsx` — серверный компонент, загружает первые 20 объявлений через Prisma
- `AdsListClient.tsx` — клиентский компонент, показывает список, поддерживает фильтры
- API `/api/tasks` — уже поддерживает offset-пагинацию (`page`, `limit`, `total`, `totalPages`)
- Sitemap включает `/ads` с `changeFrequency: "hourly"` + отдельные `/ads/{id}`
- Пагинация НЕ используется на публичной странице (только `take: 20`)

## Решение

URL-based серверная пагинация (`/ads?page=2`) + клиентские фильтры.

### 4. Обновить серверный fetch в `page.tsx`

```tsx
async function fetchInitialAds(page: number) {
  const where = {
    status: { in: ["active", "paused"] },
    deletedAt: null,
  };
  
  const [raws, total] = await Promise.all([
    prisma.ad.findMany({
      where,
      orderBy: { publishedAt: "desc" },
      skip: (page - 1) * PAGE_SIZE,
      take: PAGE_SIZE,
      include: { /* ... как сейчас */ },
    }),
    prisma.ad.count({ where }),
  ]);
  
  return {
    ads: raws.map(mapPrismaAdToAd),
    total,
    totalPages: Math.ceil(total / PAGE_SIZE),
  };
}
```

## Фасетный поиск (Faceted Search) + "Найдено X объявлений"

### Суть

Каждый фильтр показывает количество результатов рядом с каждой опцией. Выбор одного фильтра пересчитывает счётчики во всех остальных фильтрах. Если у опции 0 результатов — она disabled.

**Пример:**
```
Без фильтров:              С фильтром city=Алматы:
─────────────              ──────────────────────
Платформа                  Платформа
☐ TikTok      50           ☐ TikTok      20
☐ Instagram   60           ☐ Instagram   25
☐ YouTube     40           ☐ YouTube     15

Город                      Город
☐ Алматы      60           ☑ Алматы      60
☐ Астана      45           ☐ Астана      45   (кол-во городов НЕ зависит от выбора города)
☐ Шымкент    30           ☐ Шымкент    30

Найдено: 150               Найдено: 60
```

**Принцип:** для каждого фильтра счётчик считается с учётом ВСЕХ ДРУГИХ активных фильтров, но БЕЗ фильтра своей же размерности. Это стандартный подход faceted search.

Текстовый поиск (`search`) влияет на ВСЕ фасеты — если ввели "одежда", все счётчики пересчитываются с учётом поиска.

### Тип данных

```ts
// lib/types/ad.ts
export interface AdFacets {
  platform: Record<string, number>;    // { TikTok: 25, Instagram: 30, YouTube: 15 }
  city: Record<string, number>;        // { Almaty: 40, Astana: 20, ... } (DB enum keys)
  category: Record<string, number>;    // { KinoNarezki: 10, Memy: 5, ... } (DB enum keys)
  budgetType: Record<string, number>;  // { fixed: 30, negotiable: 20, ... }
}
```

#### 2. `app/api/tasks/route.ts` — вычислять фасеты в GET

Для каждой из 4 размерностей (platform, city, category, budgetType) запускаем `prisma.ad.groupBy()` параллельно. Каждый `groupBy` использует where-условие со ВСЕМИ фильтрами КРОМЕ своей размерности + search + базовые (status, deletedAt).

```ts
// Базовые условия (всегда применяются)
const baseWhere = {
  status: { in: ["active", "paused"] },
  deletedAt: null,
  // + search (если есть)
};

// Для platform facet: все фильтры КРОМЕ platform
const platformFacetWhere = { ...baseWhere, ...(city && { city }), ...(category && { category }), ...(budgetType && { budgetType }) };
// Для city facet: все фильтры КРОМЕ city
const cityFacetWhere = { ...baseWhere, ...(platform && { platform }), ...(category && { category }), ...(budgetType && { budgetType }) };
// ...и т.д.

const [platformCounts, cityCounts, categoryCounts, budgetTypeCounts] = await Promise.all([
  prisma.ad.groupBy({ by: ['platform'], where: platformFacetWhere, _count: true }),
  prisma.ad.groupBy({ by: ['city'],     where: cityFacetWhere,     _count: true }),
  prisma.ad.groupBy({ by: ['category'], where: categoryCounts,     _count: true }),
  prisma.ad.groupBy({ by: ['budgetType'], where: budgetTypeFacetWhere, _count: true }),
]);

// Преобразуем в Record<string, number>:
const facets = {
  platform: Object.fromEntries(platformCounts.map(r => [r.platform, r._count])),
  city: Object.fromEntries(cityCounts.map(r => [r.city, r._count])),
  category: Object.fromEntries(categoryCounts.map(r => [r.category, r._count])),
  budgetType: Object.fromEntries(budgetTypeCounts.map(r => [r.budgetType, r._count])),
};
```

Добавляем в JSON-ответ: `{ data, pagination, facets }`.

Все 4 `groupBy` + основной `findMany` + `count` = 6 запросов параллельно. `groupBy` по enum-колонкам — очень быстрый (мс).

```tsx
// Форматирование: "7 942 объявления"
function formatAdCount(n: number): string {
  const formatted = n.toLocaleString("ru-RU");
  const mod100 = n % 100;
  const mod10 = n % 10;
  let word: string;
  if (mod100 >= 11 && mod100 <= 19) word = "объявлений";
  else if (mod10 === 1) word = "объявление";
  else if (mod10 >= 2 && mod10 <= 4) word = "объявления";

}
```

## Синхронизация фильтров с URL + SSR

### Проблема

Сейчас фильтры живут только в `useState` клиента — при обновлении страницы или переходе по ссылке `/ads?city=Алматы&platform=TikTok` фильтры теряются. SSR не учитывает фильтры (кроме `page`).

### Цель

URL вида `/ads?city=Алматы&platform=TikTok&page=2` должен:
1. **SSR** — сервер читает ВСЕ query params, делает Prisma-запрос с фильтрами, считает фасеты с кросс-фильтрацией
2. **Визуально** — фильтры в sidebar отмечены как выбранные при загрузке
3. **Интерактивно** — при клике на фильтр URL обновляется + данные перезагружаются
4. **SEO** — каждая комбинация фильтров — уникальный crawlable URL

### Подход: гибрид (client-fetch + URL sync)

- Клик на фильтр → `window.history.replaceState` (обновляет URL без навигации) + client-fetch к API (быстрый UX)
- Прямой переход по URL → SSR с фильтрами (server component читает params, фетчит из Prisma)
- Поиск — URL обновляется после debounce

`replaceState` вместо `pushState` — чтобы не засорять историю при каждом клике.

### Формат URL-параметров

Фильтры в URL используют **те же строки, что отправляются в API** (русские названия для city/category, т.к. API делает `toDbCity`/`toDbCategory`):

| Параметр | Пример | Описание |
|---|---|---|
| `city` | `Алматы` | Русская строка (API конвертирует через `toDbCity`) |
| `platform` | `TikTok` | Совпадает с DB (ASCII) |
| `category` | `Мемы` | Русская строка (API конвертирует через `toDbCategory`) |
| `budgetType` | `fixed` | Совпадает с DB (ASCII) |
| `sortBy` | `budget` | "new" или "budget" |
| `search` | `одежда` | Произвольный текст |
| `page` | `2` | Номер страницы |

Пример: `/ads?city=%D0%90%D0%BB%D0%BC%D0%B0%D1%82%D1%8B&platform=TikTok&page=2`

```tsx
/** Update browser URL to reflect current filters (no navigation) */
function syncURL(filters: AdFilters, page: number) {
  const params = new URLSearchParams();
  if (filters.city) params.set("city", filters.city);
  if (filters.platform) params.set("platform", filters.platform);
  if (filters.category) params.set("category", filters.category);
  if (filters.budgetType) params.set("budgetType", filters.budgetType);
  if (filters.sortBy) params.set("sortBy", filters.sortBy);
  if (filters.search) params.set("search", filters.search);
  if (page > 1) params.set("page", String(page));

  const query = params.toString();
  const url = query ? `/ads?${query}` : "/ads";
  window.history.replaceState(null, "", url);
}
```

Вызывается:
- При изменении фильтров (в `fetchAds` после успешного запроса)
- При смене страницы
- При сбросе фильтров

**Search input** — инициализируется из `initialFilters.search`:
```tsx

```

**Пагинация** — всегда client-fetch + URL sync (убрать разделение на "SEO-режим vs client-режим"). SEOPagination всегда получает `onPageChange` callback, но URL обновляется через `syncURL` так что при перезагрузке страницы SSR отдаёт правильные данные.

Ждите — а как же SEO-ссылки (`<Link>`)? Сейчас пагинация рендерит `<Link>` когда нет фильтров. Если мы всегда используем `onPageChange`, краулеры не увидят ссылки.

**Решение:** SEOPagination строит `href` с учётом текущих фильтров. Пагинация ВСЕГДА рендерит `<Link>` для SEO. `onPageChange` используется В ДОПОЛНЕНИЕ к навигации для fast UX (intercept click, do client-fetch, then let Next.js handle the navigation).

Проще: убрать `onPageChange`, пусть SEOPagination всегда использует `<Link>`. При клике — Next.js server component re-renders (не такой быстрый, но SEO-correct). Для UX можно показать loading state через `useTransition` или `loading.tsx`.

Ещё проще: **оставить текущий подход** — `<Link>` без фильтров, `onPageChange` с фильтрами, НО `onPageChange` теперь вызывает `syncURL` тоже. При прямом переходе по URL всё работает через SSR.

Решение — оставляем как есть: SEOPagination продолжает рендерить `<Link>` для чистых URL (без фильтров), а при активных фильтрах — `onPageChange` callback + `syncURL`.

#### 3. `components/ads/SEOPagination.tsx` — hrefFor с учётом фильтров

Добавить проп `searchParams` или изменить `hrefFor` чтобы строить URL с учётом текущих фильтров для `<Link>` режима. Это нужно чтобы при пагинации без `onPageChange` (SEO-режим) ссылки включали фильтры.

Нет, при чистом SEO-режиме (без фильтров) `hrefFor` уже работает: `/ads?page=2`. А при фильтрах используется `onPageChange`. Менять не нужно.

## Мультиселект + DB-коды в URL + сохранение фильтров при возврате

### 3 задачи

1. **Мультиселект** — можно выбрать несколько значений в каждом фильтре (TikTok + Instagram, Алматы + Астана)

### Сохранение фильтров при возврате — ИСПРАВЛЕНИЕ

**Проблема:** `window.history.replaceState` обновляет URL в браузере, но **Next.js App Router не знает об этом изменении**. Когда пользователь нажимает "назад", Next.js восстанавливает свою закешированную версию страницы (без фильтров в searchParams), а не URL из `replaceState`.

**Две кнопки "назад" не работают:**
1. `<Link href="/ads">` в `AdDetailClient.tsx` — хардкодит `/ads` без params
2. Browser back button — Next.js cache не знает про `replaceState`

**Решение:**

#### 1. `app/ads/AdsListClient.tsx` — заменить `replaceState` на `router.replace`

```tsx
import { useRouter } from "next/navigation";

const router = useRouter();

// Было:
window.history.replaceState(null, "", url);

// Станет:
router.replace(url, { scroll: false });
```

`router.replace` обновляет Next.js router cache — при back-навигации Next.js знает правильный URL с searchParams и ре-рендерит server component с фильтрами.

`{ scroll: false }` — не скроллит наверх при обновлении URL.

**Важно:** `router.replace` триггерит server component re-render (SSR). Но клиент уже имеет актуальные данные от client-fetch. `useState` не переинициализируется при ре-рендере (сохраняет текущее состояние). Поэтому данные не "мигают".

Функция `syncURL` становится методом внутри компонента (нужен доступ к `router`):
```tsx
const syncURL = useCallback((f: AdFilters, page: number) => {
  const params = new URLSearchParams();
  if (f.city?.length) params.set("city", f.city.join(","));
  // ...
  const qs = params.toString();
  const url = qs ? `/ads?${qs}` : "/ads";
  router.replace(url, { scroll: false });
}, [router]);
```

#### 2. `app/ads/[id]/AdDetailClient.tsx` — "Назад" через `router.back()`

Сейчас:
```tsx
<Link href="/ads" className="...">
  <ArrowLeftOutlined />
  Назад / Объявления
</Link>
```

Станет:
```tsx
<button type="button" onClick={() => router.back()} className="...">
  <ArrowLeftOutlined />
  Назад / Объявления
</button>
```

`router.back()` возвращает на предыдущую запись history — `/ads?city=Almaty&platform=TikTok`. Server component ре-рендерится с searchParams → SSR с фильтрами → клиент инициализируется правильно.

Вторая ссылка (`<Link href="/ads">` для "Смотреть другие объявления" при неактивном объявлении) остаётся как есть — это не "назад", а навигация в каталог.

## Сохранение страницы пагинации при возврате

### Проблема

Пользователь на странице 2 → открывает объявление → возвращается назад → видит страницу 1 (а не 2).

**Корневая причина** в `AdsListClient.tsx`:

```tsx
// Строки 177-181:
useEffect(() => {
  setClientPage(1);     // ← ВСЕГДА сбрасывает на 1
  fetchAds(1);          // ← ВСЕГДА фетчит page 1
}, [fetchAds]);
```

`fetchAds` пересоздаётся при каждом маунте (зависимости `[filters, pageSize, router]`). При back-навигации компонент ре-маунтится → `fetchAds` новая ссылка → `useEffect` срабатывает → **сбрасывает на page 1**.

Внутри `fetchAds` есть guard `isFirstRender.current` который пропускает fetch, но `setClientPage(1)` в `useEffect` выполняется ДО `fetchAds`, и это уже меняет состояние.

### Решение

Убрать `setClientPage(1)` из `useEffect` и перенести сброс страницы в саму логику изменения фильтров. `useEffect` должен вызывать `fetchAds` с текущей `clientPage`, а не хардкодить 1.

Но нужно различать два случая:
1. **Фильтры изменились** (юзер кликнул фильтр) → сбросить на page 1
2. **Первый рендер / back-навигация** (фильтры из URL, SSR данные актуальны) → НЕ сбрасывать

**Подход:** вместо `useEffect` на `fetchAds`, отслеживать изменение фильтров явно через ref:

```tsx
const prevFiltersRef = useRef<AdFilters>(initialFilters ?? {});

useEffect(() => {
  // При первом рендере — SSR данные актуальны, не фетчим
  if (isFirstRender.current) {
    isFirstRender.current = false;
    return;
  }

  // Фильтры изменились → сбросить на page 1 и перефетчить
  setClientPage(1);
  fetchAdsRaw(1);  // fetchAdsRaw — без isFirstRender guard
}, [filters]);  // Зависимость только от filters
```

Или ещё проще: перенести `isFirstRender` guard ИЗ `fetchAds` В `useEffect`:

```tsx
const isFirstRender = useRef(true);

useEffect(() => {
  if (isFirstRender.current) {
    isFirstRender.current = false;
    return;  // SSR data is fresh — skip fetch and keep page from props
  }
  // Фильтры изменились → reset to page 1
  setClientPage(1);
  doFetch(1);
}, [filters, sortBy, search...]);  // НЕ зависим от fetchAds
```

Но `doFetch` нужен стабильный (не зависящий от `filters`, иначе будет бесконечный цикл).

**Лучший подход:**

1. Убрать `isFirstRender` guard из `fetchAds`
2. Перенести skip-логику в `useEffect`
3. `fetchAds` принимает и фильтры и page как аргументы (не читает из state)

```tsx
// fetchAds теперь принимает всё как аргументы — стабильная функция
const fetchAds = useCallback(
  async (f: AdFilters, page: number) => {
    setLoading(true);
    try {
      const params = new URLSearchParams();
      if (f.city?.length) params.set("city", f.city.join(","));
      // ...build params from f...
      params.set("page", String(page));
      params.set("limit", String(pageSize));
      
      const res = await fetch(`/api/tasks?${params.toString()}`);
      // ...process response...
      
      setAds(adapted);
      setClientPage(page);
      // ...
      router.replace(buildURL(f, page), { scroll: false });
    } catch { setAds([]); }
    finally { setLoading(false); }
  },
  [pageSize, router],  // Стабильные зависимости — НЕ зависит от filters
);

// useEffect — реагирует на изменение фильтров
const isFirstRender = useRef(true);

useEffect(() => {
  if (isFirstRender.current) {
    isFirstRender.current = false;
    return;  // SSR data matches URL — skip fetch, keep currentPage
  }
  // Фильтры изменились → page 1
  fetchAds(filters, 1);
}, [filters, fetchAds]);

// Пагинация — явный вызов с текущими фильтрами и нужной страницей
const handlePageChange = (page: number) => {
  fetchAds(filters, page);
};
```

**Ключевые отличия:**
- `fetchAds` стабильная (зависит только от `pageSize`, `router`) — `useEffect` не триггерится при ре-маунте
- `useEffect` зависит от `filters` — триггерится ТОЛЬКО при реальном изменении фильтров
- `isFirstRender` guard в `useEffect` пропускает первый рендер (SSR данные уже правильные)
- При back-навигации: компонент маунтится с `initialFilters` из URL → `filters` = `initialFilters` → `useEffect` видит `isFirstRender=true` → пропускает → `clientPage` остаётся `currentPage` из SSR props → страница 2 сохраняется
