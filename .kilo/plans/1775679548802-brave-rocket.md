### Что делает seed-скрипт

1. Для каждого из 10 профилей:
   - Находит профиль по `fullName`
   - Создаёт N юзеров-ревьюверов (уникальные email, например `reviewer-1-asyl@seed.kz`)
   - Создаёт `ContactInteraction` для каждого reviewer↔profile (для консистентности)
   - Создаёт `Review` с `reviewerId`, `targetType: "creator_profile"`, `targetId`, `creatorProfileId`, `rating`, `comment`
2. После создания всех отзывов — пересчитывает `averageRating` и `reviewCount` через `recalculateCreatorRating()`

# Fix: 500 ошибка при пагинации каталога креаторов

## Причина

`CREATOR_INCLUDE` определён на уровне модуля (вне функции-обработчика) и содержит:

```ts
boosts: { where: { expiresAt: { gte: new Date() } } }
```

`new Date()` вычисляется **один раз** при загрузке модуля. С PrismaPg driver adapter (v7.5) это может вызывать ошибку сериализации Date-объекта при повторных запросах, потому что адаптер ожидает «свежий» Date.

## Решение

Заменить статический `CREATOR_INCLUDE` на функцию `getCreatorInclude()` которая возвращает объект с `new Date()` при каждом вызове.

## Файлы для изменения (4 файла)

### 1. `app/api/creators/route.ts`

**Было:**
```ts
const CREATOR_INCLUDE = {
  platforms: true,
  portfolio: { orderBy: { createdAt: "desc" as const } },
  priceItems: { orderBy: { sortOrder: "asc" as const } },
  boosts: { where: { expiresAt: { gte: new Date() } } },
  user: { select: { id: true, name: true, email: true, avatarColor: true } },
};
```

**Стало:**
```ts
function getCreatorInclude() {
  return {
    platforms: true,
    portfolio: { orderBy: { createdAt: "desc" as const } },
    priceItems: { orderBy: { sortOrder: "asc" as const } },
    boosts: { where: { expiresAt: { gte: new Date() } } },
    user: { select: { id: true, name: true, email: true, avatarColor: true } },
  };
}
```

И использовать: `include: getCreatorInclude()` вместо `include: CREATOR_INCLUDE`.

### 2. `app/api/creators/[id]/route.ts`
Та же замена `const CREATOR_INCLUDE` → `function getCreatorInclude()`.

### 3. `app/api/creators/[id]/preview/route.ts`
Та же замена.

### 4. `app/api/creators/my/route.ts`
Вместо статического `include` — инлайн boosts filter:
```ts
include: {
  platforms: true,
  portfolio: { orderBy: { createdAt: "desc" } },
  priceItems: { orderBy: { sortOrder: "asc" } },
  boosts: { where: { expiresAt: { gte: new Date() } } },
```

# Fix: Бустнутые кластерами, а не перемешаны

## Проблема

Сортировка `raisedAt DESC` создаёт кластеры: Premium всегда #1-3, все обычные в середине, VIP и Rise в конце. На реальных площадках (OLX, Kolesa, Krisha) бустнутые **равномерно распределены** среди обычных.

## Решение: Post-query interleaving

После получения результата из Prisma — **перемешать** бустнутые среди обычных на уровне JavaScript. Не менять запрос к БД, менять только порядок в ответе.

### Алгоритм `interleaveBoosts(items)`

```
Input:  [P, P, P, R1, R2, R3, R4, R5, R6, R7, R8, R9, R10, R11, R12, R13, R14, R15, R16, R17]
        (3 boosted + 17 regular, sorted by raisedAt)

1. Разделяем: boosted = [P, P, P], regular = [R1..R17]
2. Рассчитываем шаг: step = floor(total / (boosted.length + 1))
   - 20 / (3 + 1) = 5 → вставляем бустнутый каждые 5 позиций
3. Вставляем:
   - pos 0: R1        pos 5: [PREMIUM]   pos 10: [PREMIUM]   pos 15: [PREMIUM]
   - Остальные позиции: regular items

Output: [R1, R2, R3, R4, R5, P, R6, R7, R8, R9, P, R10, R11, R12, R13, P, R14, R15, R16, R17]
```

Бустнутые равномерно распределены среди обычных на каждой странице.

### Функция (shared helper)

```ts
// lib/interleave-boosts.ts
export function interleaveBoosts<T extends { boosts?: unknown[] }>(items: T[]): T[] {
  if (items.length === 0) return items;

  const boosted: T[] = [];
  const regular: T[] = [];

  for (const item of items) {
    if (item.boosts && item.boosts.length > 0) {
      boosted.push(item);
    } else {
      regular.push(item);
    }
  }

  if (boosted.length === 0 || regular.length === 0) return items;

  const result: T[] = [...regular];
  const step = Math.max(1, Math.floor(result.length / (boosted.length + 1)));

  for (let i = 0; i < boosted.length; i++) {
    const insertAt = Math.min((i + 1) * step + i, result.length);
    result.splice(insertAt, 0, boosted[i]);
  }

  return result;
}
```

#### 2. `app/api/creators/route.ts` — API route

Перед `return NextResponse.json(...)` вызываем interleave. Prisma query остаётся как есть (raisedAt DESC обеспечивает что бустнутые попадают на ранние страницы), но внутри каждой страницы они перемешиваются.

# Fix v2: Бустнутые кластерами на последних страницах

## Проблема

`interleaveBoosts()` перемешивает только ВНУТРИ одной страницы. Но если VIP/Rise ещё не подняты cron'ом, они попадают на последнюю страницу целым кластером (9 из 15).

**Корень**: один запрос сортирует по `raisedAt DESC` → бустнутые с неактуальным `raisedAt` скатываются в конец.

## Решение: Два отдельных запроса + распределение по страницам

Вместо одного `findMany` делаем **два**: для бустнутых и для обычных. Рассчитываем сколько бустнутых слотов на каждую страницу, и заполняем остальное обычными. Потом `interleaveBoosts()` перемешивает внутри страницы.

### Алгоритм

```
totalAll = 55, totalBoosted = 12, totalRegular = 43
totalPages = ceil(55 / 20) = 3
boostedPerPage = ceil(12 / 3) = 4
regularPerPage = 20 - 4 = 16

Page 1: 4 boosted (skip 0) + 16 regular (skip 0)  = 20  → interleave
Page 2: 4 boosted (skip 4) + 16 regular (skip 16) = 20  → interleave
Page 3: 4 boosted (skip 8) + 11 regular (skip 32) = 15  → interleave
```

Каждая страница получает ~4 бустнутых профиля, равномерно распределённых.

### Prisma фильтры

```ts
const now = new Date();

// Бустнутые = есть хотя бы один активный буст
const boostedWhere = { ...where, boosts: { some: { expiresAt: { gte: now } } } };

// Обычные = нет ни одного активного буста
const regularWhere = { ...where, NOT: { boosts: { some: { expiresAt: { gte: now } } } } };
```

### Файлы для изменения (2 файла)

#### 1. `app/api/creators/route.ts`

Заменить один `findMany` на два + подсчёт бустнутых:


# Fix v3: Бустнутые сортируются по приоритету (premium первый)

## Проблема

Бустнутые профили сортируются по `createdAt DESC` — порядок зависит от даты создания, а не от типа буста. В результате Rise (дешёвый) может стоять выше Premium (дорогой) на странице.

## Решение

Сортировка бустнутых: сначала по **приоритету буста** (premium → vip → rise), затем по `createdAt DESC`.

В Prisma нельзя сортировать по вложенному полю `boosts.boostType` напрямую. Поэтому применяем **post-query сортировку** в `interleaveBoosts()` — перед вставкой бустнутых сортируем их по приоритету.

### Изменение в `lib/interleave-boosts.ts`

Добавить сортировку `boosted[]` по приоритету перед вставкой:

```ts
const BOOST_PRIORITY: Record<string, number> = { premium: 3, vip: 2, rise: 1 };

function getBoostPriority(item: { boosts?: unknown[] | null }): number {
  if (!item.boosts || item.boosts.length === 0) return 0;
  let max = 0;
  for (const b of item.boosts) {
    const bt = typeof b === "string" ? b : (b as { boostType?: string }).boostType;
    max = Math.max(max, BOOST_PRIORITY[bt ?? ""] ?? 0);
  }
  return max;
}

// В функции interleaveBoosts, после разделения на boosted/regular:
boosted.sort((a, b) => getBoostPriority(b) - getBoostPriority(a));
```

Это даёт: Premium вставляется первым (ближе к началу страницы), VIP — в середине, Rise — ближе к концу.

### Результат

```
Page 1: [R, R, R, PREMIUM, R, R, R, VIP, R, R, R, VIP, R, R, R, RISE, R, R, R, R]
Page 2: [R, R, R, PREMIUM, R, R, R, VIP, R, R, R, RISE, R, R, R, RISE, R, R, R, R]
Page 3: [R, R, PREMIUM, R, R, VIP, R, R, RISE, R, R, RISE, R, R, R]
```

Premium всегда на лучшей позиции страницы → платишь больше — получаешь больше.

# Fix v4: Бустнутые должны быть НАВЕРХУ страницы, а не в середине

## Исследование: как РЕАЛЬНО работают бусты

Из официальной документации Avito:
> "Объявление будет на разных, в том числе первых, позициях в поиске"
> "Если вам важно показываться на первых строчках как можно чаще, используйте самую сильную услугу"

**Вывод**: бустнутые элементы показываются НАВЕРХУ страницы. Это то, за что платят деньги — видимость в верхних позициях. Не в середине, не внизу.

## Что сейчас неправильно

`interleaveBoosts()` ставит бустнутых на позиции 4, 8, 12, 16 — в СЕРЕДИНУ страницы.
Человек платит 3990₸ за Premium, а получает 4-ю позицию вместо 1-й.

## Правильное поведение

На каждой странице:
1. **Сначала бустнутые** — Premium первый, VIP вторым, Rise третьим
2. **Потом обычные** — по выбранной сортировке (createdAt, price, rating, popular)
3. Распределение по страницам — **round-robin** (уже реализован)

```
Page 1: [PREMIUM₁, VIP₁, RISE₁, RISE₂, regular₁, regular₂, ..., regular₁₆]
Page 2: [PREMIUM₂, VIP₂, RISE₃, RISE₄, regular₁₇, regular₁₈, ..., regular₃₂]
```

## Итоговое поведение

- Premium платит 3990₸ → позиция #1 на странице, появляется на каждой странице
- VIP платит 2990₸ → позиция #2 на странице
- Rise платит 1990₸ → позиция #3-4 на странице
- Обычные → позиции #5-20

Это стандартная модель Avito/OLX/Kolesa — платишь за верхние позиции.

### 4. Ротация внутри одного тира (справедливое распределение)

Если 10 человек купили Premium — каждый должен получить равные шансы быть #1.

Обновить `sortByBoostPriority()` в `lib/interleave-boosts.ts`:
1. Группируем по приоритету (premium/vip/rise)
2. **Шаффлим** элементы внутри группы с seed-ом привязанным к текущему часу
3. Конкатенируем группы обратно

```ts
const hourSeed = Math.floor(Date.now() / (60 * 60 * 1000));

function seededShuffle<T>(arr: T[], seed: number): T[] {
  const result = [...arr];
  let s = seed;
  for (let i = result.length - 1; i > 0; i--) {
    s = (s * 1103515245 + 12345) & 0x7fffffff;
    const j = s % (i + 1);
    [result[i], result[j]] = [result[j], result[i]];
  }
  return result;
}

export function sortByBoostPriority<T>(items: T[]): T[] {
  const groups: Record<number, T[]> = {};
  for (const item of items) {
    const p = getBoostPriority(item);
    (groups[p] ??= []).push(item);
  }
  // Группы по убыванию приоритета, внутри — часовой шаффл
  return [3, 2, 1]
    .flatMap(p => seededShuffle(groups[p] ?? [], hourSeed + p));
}
```

**Результат**: каждый час порядок внутри Premium/VIP/Rise меняется. Все Premium'ы по очереди бывают на позиции #1. Пагинация стабильна в пределах часа (один и тот же seed).

# Применить ту же логику буст-распределения к объявлениям (ads)

## Текущее состояние ads

Объявления используют **старую логику** — один `findMany` с `raisedAt DESC`:
- `app/api/tasks/route.ts:117-122` — сортировка `[{ raisedAt: desc, nulls: last }, secondary]`
- `app/ads/page.tsx:240-249` — SSR страница, та же сортировка

**Проблема**: все бустнутые объявления кластерятся наверху страницы 1, на остальных страницах их нет. Та же проблема что была у креаторов.

## Решение

Применить точно такой же паттерн как для креаторов:

1. **Два запроса**: boosted (с активными бустами) + regular (без бустов)
2. **Round-robin**: `ceil(totalBoosted / totalPages)` бустнутых на страницу
3. **sortByBoostPriority()**: Premium наверху, с часовой ротацией внутри тира
4. **Конкатенация**: `[...boostedItems, ...regularItems]`

### Отличия ads от creators

| | Creators | Ads |
|--|----------|-----|
| Модель буста | `CreatorBoost` | `AdBoost` |
| Include | `boosts: { where: { expiresAt: { gte: now } } }` | `boosts: { where: { expiresAt: { gt: now } }, select: { boostType: true, expiresAt: true } }` |
| Secondary sort | createdAt/price/rating/popular | publishedAt/rpm/budget |
| Клиентская пагинация | `CreatorsListClient.tsx` | `AdsListClient.tsx` |

### Файлы для изменения (2 файла)

#### 1. `app/api/tasks/route.ts`

Изменения аналогичны `app/api/creators/route.ts`:

1. Добавить `import { sortByBoostPriority } from "@/lib/interleave-boosts";`
2. Убрать `raisedAt` из `orderBy` — оставить только secondary sort
3. Разделить `where` на `boostedWhere` и `regularWhere`:
   ```ts
   const now = new Date();
   const boostedWhere = { ...where, boosts: { some: { expiresAt: { gt: now } } } };
   const regularWhere = { ...where, NOT: { boosts: { some: { expiresAt: { gt: now } } } } };
   ```
   (Ads используют `gt` вместо `gte` в `boosts` include)
4. Подсчёт `total` и `totalBoosted` параллельно с фасетами
5. Расчёт слотов: `boostedPerPage`, `boostedSliceStart/End`, `regularSkip/Take`
6. Два запроса: `allBoosted` (без пагинации) + `regularItems` (с пагинацией)
7. `sortByBoostPriority(allBoosted).slice(...)` + `[...boostedItems, ...regularItems]`

#### 2. `app/ads/page.tsx`

То же самое для SSR:

1. Добавить `import { sortByBoostPriority } from "@/lib/interleave-boosts";`
2. Убрать `raisedAt` из `orderBy`
3. Два запроса: boosted + regular
4. `sortByBoostPriority()` + конкатенация
