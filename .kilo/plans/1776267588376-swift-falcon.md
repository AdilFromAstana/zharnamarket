# Plan: Incremental Refactor of Static Categories/Cities to Database-Driven CRUD

## Overview
The current system uses static constants and enums for categories, cities, and platforms. UI sends Russian labels but API expects English enum keys, causing validation failures. We need to migrate to database tables incrementally while preserving functionality.

## Current Issues
- UI sends Russian labels (e.g., "Мемы") but validation expects enum keys (e.g., "Memy")
- Static data scattered across codebase prevents dynamic management
- No API endpoints for fetching reference data
- Inflexible approach limits scalability

## Proposed Solution
Gradual migration: inventory static usages → seed database → replace with API calls → remove static data.

## Database Tables Design

model Category {
  id          String @id @default(cuid())
  key         String @unique // English key for internal use (e.g., "Memy")

  sortOrder   Int // For ordering in UI
  isActive    Boolean @default(true) // To enable/disable categories

  // Relations
  creatorProfiles CreatorProfile[]

## Steps

### Stage 1: Inventory Static Data Usage
- **Backend**: Find all references to VALID_CATEGORIES, VALID_CITIES, VALID_PLATFORMS in validation.ts and API routes
- **Frontend**: Find all imports/usage of CATEGORIES, CITIES, PLATFORMS, BUDGET_TYPES, BUSINESS_CATEGORIES from lib/constants.ts
- **Collect into unified arrays**: Create comprehensive lists of all static reference data with mappings (Russian label → English key)
- **Output**: Document all locations and create mapping objects

### Stage 2: Create Database Tables and Seed Data
- **Prisma Schema**: Add Category and City models with fields: id, key (unique), label, sortOrder, isActive
- **Migration**: Create tables (keep existing enums temporarily for compatibility)
- **Seed Script**: Populate tables with current static data if not exists
- **Mapping**: Ensure Russian labels map to English keys for backward compatibility

### Stage 3: Replace Static Validation with Database Queries
- **Update Validation Logic**: Replace VALID_CATEGORIES/VALID_CITIES sets with DB queries
- **API Routes**: Update creators and ads endpoints to validate against DB data
- **Mapping Functions**: Add functions to convert Russian labels to DB keys during API processing
- **Preserve Functionality**: Ensure existing API behavior works unchanged

### Stage 4: Create GET API Endpoints
- **Categories API** (`/api/categories`): GET endpoint returning active categories with labels
- **Cities API** (`/api/cities`): GET endpoint returning active cities with labels
- **Platforms API** (`/api/platforms`): GET endpoint (platforms are simpler, may stay static)
- **Budget Types API** (`/api/budget-types`): GET endpoint for budget types
- **Business Categories API** (`/api/business-categories`): GET endpoint

### Stage 5: Update Frontend to Use API
- **Replace Constants**: Update lib/constants.ts to export functions that fetch from API
- **Update Components**: Modify forms in creators/new, ads/new to fetch data via API calls
- **Loading States**: Add proper loading/error handling for async data fetching
- **Backward Compatibility**: Ensure components work during transition

### Stage 6: Remove Static Data and Enums
- **Clean Up**: Remove static constants from lib/constants.ts and validation.ts
- **Prisma Schema**: Remove ContentCategory and City enums after migration
- **Update Models**: Change CreatorProfile and Ad models to use relations instead of enums
- **Final Migration**: Migrate existing data to use new table relations

### Stage 7: Add CRUD Operations
- **Categories API**: Add POST/PUT/DELETE for category management
- **Cities API**: Add POST/PUT/DELETE for city management
- **Admin Interface**: Consider adding admin routes for managing reference data

### Stage 8: Testing and Cleanup
- **Unit Tests**: Test validation functions with DB queries
- **Integration Tests**: Test creator/ad creation with new system
- **E2E Tests**: Verify UI loads data correctly and forms submit successfully
- **Performance**: Ensure API calls don't impact page load times

## Benefits
- Eliminates validation errors with Russian labels
- Enables dynamic management of categories/cities
- Centralized reference data management
- Better maintainability and scalability

## Risks
- Complex transition requiring careful coordination
- Potential temporary inconsistencies during migration
- Need for comprehensive testing to ensure no breakage
