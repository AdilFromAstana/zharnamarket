# Фикс: middleware.ts → proxy.ts конфликт

## Проблема

Next.js 16 заменил `middleware.ts` на `proxy.ts`. Проект уже имеет `proxy.ts`
для защиты маршрутов. Созданный ранее `middleware.ts` вызывает конфликт:
```
Error: Both middleware file "./middleware.ts" and proxy file "./proxy.ts" are detected.
```

## Решение

1. **Удалить `middleware.ts`** — он несовместим с `proxy.ts`
2. **Добавить логику visitor cookie в существующий `proxy.ts`**

Текущий matcher в `proxy.ts` исключает API-роуты (`(?!api|...)`), но это нормально:
- Cookie `vw_vid` ставится с `path="/"` при SSR-визите на `/ads` или `/creators`
- API-роуты (`/api/tasks`, `/api/creators`) просто читают уже установленную cookie
- `visitorId` опционален — если cookie ещё нет, fallback на hourSeed only

## Файлы для изменения

### 1. УДАЛИТЬ `middleware.ts`

### 2. Обновить `proxy.ts` — добавить visitor cookie

В конце функции `proxy()`, перед `return NextResponse.next()`, добавить установку cookie:

```typescript
const VISITOR_COOKIE = "vw_vid";
const ONE_YEAR = 365 * 24 * 60 * 60;

export function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl;
  // ... существующая auth логика ...

  // Visitor ID для per-visitor ротации бустов
  const response = NextResponse.next();
  if (!request.cookies.get(VISITOR_COOKIE)) {
    response.cookies.set(VISITOR_COOKIE, crypto.randomUUID(), {
      httpOnly: false,
      secure: process.env.NODE_ENV === "production",
      sameSite: "lax",
      path: "/",
      maxAge: ONE_YEAR,
    });
  }
  return response;
}
```

Важно: cookie ставится только на `NextResponse.next()`, не на redirect-ответах.

## Порядок работы

1. Удалить `middleware.ts`
2. Обновить `proxy.ts` — добавить visitor cookie логику
3. Проверить `tsc --noEmit`
