# Plan: Escrow Application Flow — Полный Редизайн

## Итоговое понимание

Два типа объявлений остаются разными:

| Тип | Поведение |
|-----|-----------|
| **`direct`** ("Через заказчика") | Без изменений — контакты видны, люди договариваются сами |
| **`escrow`** ("Через платформу") | Полный редизайн пайплайна (описан ниже) |

---

## Новый Escrow Flow — Диаграмма состояний

```
[Объявление активно]
        │
        ▼ Креатор нажимает "Откликнуться"
[TaskApplication: pending_video]  ← videoDeadline = now + 48h
        │
        ├── 48h истекло без видео ──► [video_timed_out] (авто, lazy)
        │
        ▼ Креатор прикрепляет ссылку на видео (до 48h)
[TaskApplication: pending_review]  ← contentReviewDeadline = now + 72h
        │
        ├── 72h истекло без ответа заказчика ──► [content_approved] (авто, lazy)
        ├── Заказчик нажимает "Отклонить" ──► [content_rejected]
        │
        ▼ Заказчик нажимает "Одобрить"
[TaskApplication: content_approved]
        │
        ▼ Креатор прикрепляет скриншот статистики + claimed views
[VideoSubmission: submitted] ← slaDeadline = now + 24h (существующая логика)
        │
        ├── Admin отклоняет ──► [VideoSubmission: rejected] → escrow возвращается
        │
        ▼ Admin подтверждает views
[VideoSubmission: approved]
        │
        ▼ Расчёт: (views / 1000) × RPM × 0.8 → Creator balance
[Выплата зачислена] → Вывод по существующим правилам (мин. 500₸)
```

## 1. Изменения в Prisma Schema

### 1.1 Новый enum `ContentStatus`

```prisma
enum ContentStatus {
  pending_video    // ждёт ссылку на видео (48h)
  video_timed_out  // 48h истекло — заявка отменена
  pending_review   // видео прикреплено, ждёт одобрения заказчика (72h)
  content_approved // одобрено заказчиком (или авто-одобрено)
  content_rejected // отклонено заказчиком
}
```

### 1.2 Обновление модели `TaskApplication`

Добавить поля:

```prisma
model TaskApplication {
  id        String        @id @default(cuid())
  adId      String
  ad        Ad            @relation(...)
  creatorId String
  creator   User          @relation(...)
  createdAt DateTime      @default(now())

  // ── НОВЫЕ ПОЛЯ ────────────────────────────
  contentStatus        ContentStatus @default(pending_video)
  videoDeadline        DateTime?     // createdAt + 48h — дедлайн прикрепить видео
  videoUrl             String?       // ссылка на опубликованное видео
  videoSubmittedAt     DateTime?
  contentReviewDeadline DateTime?    // videoSubmittedAt + 72h — дедлайн заказчику
  contentReviewedAt    DateTime?     // когда заказчик принял решение
  contentRejectionNote String?       // причина отклонения от заказчика
  // ──────────────────────────────────────────

  submission VideoSubmission?

  @@unique([adId, creatorId])
  @@map("task_applications")
}
```

## 2. API — Обновлённые и Новые Маршруты

### 2.1 `POST /api/tasks/[id]/apply` — ОБНОВИТЬ

**Было:** просто создаёт `TaskApplication`  
**Стало:**
- Устанавливает `contentStatus: "pending_video"`
- Устанавливает `videoDeadline = now + 48h`
- Проверяет эскроу активен и бюджет > 0 (как сейчас)

### 2.2 `POST /api/tasks/[id]/apply/video` — **НОВЫЙ**

Креатор прикрепляет ссылку на видео.

```
Тело: { videoUrl: string }

Валидация:
- Есть TaskApplication для этого creatorId
- contentStatus === "pending_video"
- Lazy check: if (now > videoDeadline) → обновить до "video_timed_out" → вернуть ошибку
- videoUrl валидный URL

Действие:
- Обновить application:
    videoUrl = body.videoUrl
    videoSubmittedAt = now
    contentReviewDeadline = now + 72h
    contentStatus = "pending_review"
```

### 2.3 `PATCH /api/tasks/[id]/applications/[appId]/review` — **НОВЫЙ**

Заказчик одобряет/отклоняет видео.

```
Тело: { action: "approve" | "reject", note?: string }

Валидация:
- Caller === ad.ownerId
- Application существует и принадлежит этому ad
- contentStatus === "pending_review"
- Lazy check: if (now > contentReviewDeadline) → уже авто-одобрено → вернуть статус

Действие при "approve":
- contentStatus = "content_approved"
- contentReviewedAt = now

Действие при "reject":
- contentStatus = "content_rejected"
- contentRejectionNote = body.note
- contentReviewedAt = now
```

### 2.4 `GET /api/tasks/[id]/applications` — **НОВЫЙ**

Заказчик получает список всех заявок на своё задание.

```
Доступ: только ad.ownerId

Возвращает: все TaskApplication с полями:
- id, contentStatus, createdAt, videoDeadline
- videoUrl, videoSubmittedAt, contentReviewDeadline
- creator: { id, name, avatar, creatorProfile }
- submission: { status, claimedViews, approvedViews } — если есть
```

### 2.5 `GET /api/tasks/[id]/apply` — **НОВЫЙ** (для текущего пользователя)

Проверяет статус своей заявки на задание.

```
Возвращает: TaskApplication или null
Lazy check: if deadline прошёл → обновить статус → вернуть актуальный
```

### 2.6 `POST /api/tasks/[id]/submissions` — ОБНОВИТЬ

Добавить проверку: `application.contentStatus === "content_approved"` (иначе — ошибка).

## 5. Обработка таймаутов — Lazy Evaluation

Вместо cron-джоба: статусы обновляются лениво при каждом API-запросе.

**Утилита `checkApplicationDeadlines(application)`:**

```typescript
function resolveApplicationStatus(app: TaskApplication): ContentStatus {
  if (app.contentStatus === "pending_video" && app.videoDeadline && new Date() > app.videoDeadline) {
    return "video_timed_out";
  }
  if (app.contentStatus === "pending_review" && app.contentReviewDeadline && new Date() > app.contentReviewDeadline) {
    return "content_approved"; // авто-одобрение
  }
  return app.contentStatus;
}
```

При каждом GET/PATCH запросе — проверить и обновить в БД если нужно.

# Plan: Универсальный кошелёк пользователя (UserWallet)

## Контекст и текущее состояние

| Тип пользователя | Сейчас |
|---|---|
| **Рекламодатель** | Платит per-action через провайдер (Kaspi/mock). Кошелька нет. |
| **Креатор** | `CreatorBalance` — зарабатывает на видео, выводит вручную |

**Задача:** единый кошелёк для любого пользователя. Пополняешь через Kaspi — тратишь на любые действия. Прямые платежи через провайдер остаются параллельно.

---

## Решения по ключевым вопросам

| Вопрос | Решение |
|---|---|
| Одна модель или по ролям? | **Одна** — расширяем `CreatorBalance` на всех пользователей |
| Оплата эскроу | Из кошелька (`balance -= totalBudget → EscrowAccount`) |
| Кошелёк + прямые платежи | **Параллельно** — пользователь выбирает способ |
| Недостаточно средств | Блокировать действие, предложить пополнить на недостающую сумму |

## Способ создания

Использовать `POST /api/auth/register` напрямую через Prisma seed скрипт:

```ts
// prisma/seed-test-users.ts
import bcrypt from "bcryptjs";
import { prisma } from "../lib/prisma";

async function main() {
  const hash = (p: string) => bcrypt.hash(p, 10);

  // Креатор
  await prisma.user.upsert({
    where: { email: "creator@test.local" },
    create: {
      name: "Test Creator",
      email: "creator@test.local",
      password: await hash("Creator123!"),
      role: "user",
      emailVerified: true,
    },
    update: {},
  });

  // Рекламодатель
  await prisma.user.upsert({
    where: { email: "advertiser@test.local" },
    create: {
      name: "Test Advertiser",
      email: "advertiser@test.local",
      password: await hash("Advertiser123!"),
      role: "user",
      emailVerified: true,
    },
    update: {},
  });

  console.log("✅ Test users created");
}

main().catch(console.error).finally(() => prisma.$disconnect());
```
