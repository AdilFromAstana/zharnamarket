# Полный аудит проекта Viral Wall — все процессы, кейсы и пробелы

## Обзор архитектуры

- **Тип**: Next.js Full-Stack (App Router) — это И фронтенд, И бэкенд
- **БД**: PostgreSQL через Prisma ORM
- **Аутентификация**: Custom JWT (HS256, jose) + httpOnly cookies
- **UI**: Ant Design + Tailwind CSS
- **Роли**: `user` | `admin` (нет отдельной роли "creator")
- **Валюта**: KZT (тенге), рынок Казахстана
- **Платформы**: TikTok, Instagram, YouTube
- **45 API-роутов**, ~30 страниц

## 1. РЕГИСТРАЦИЯ И ВХОД

### Сценарий 1.1: Регистрация нового пользователя

| Шаг | Что происходит | Статус |
|-----|---------------|--------|
| Заходит на `/auth/register` | Форма: имя, телефон, email, пароль, галочка согласия | ГОТОВО |
| Заполняет форму | Валидация: имя 1-100, email формат, пароль 8-128, телефон 7-20 цифр | ГОТОВО |
| Нажимает "Зарегистрироваться" | POST `/api/auth/register`, rate limit 5/час на IP | ГОТОВО |
| Бэкенд обрабатывает | Проверка уникальности email, хеширование пароля bcrypt(12), создание пользователя | ГОТОВО |
| **После регистрации** | **Автоматический вход** — JWT токены + cookies выставляются сразу | ГОТОВО |
| Редирект | Перенаправление на `/` (главная) | ГОТОВО |

**Пробелы:**
- **Нет подтверждения email** — пользователь регистрируется с любым email без верификации
- **Несоответствие телефона** — фронтенд требует телефон, бэкенд принимает без него
- **"Запомнить меня"** — галочка есть на UI, но ничего не делает (токены всегда одинаковые)

### Сценарий 1.2: Вход по логину/паролю

| Шаг | Что происходит | Статус |
|-----|---------------|--------|
| Заходит на `/auth/login` | Форма: email или телефон + пароль | ГОТОВО |
| Вводит данные | Поиск по email ИЛИ phone в БД | ГОТОВО |
| Неверный пароль | Ошибка, rate limit 10/15мин на IP | ГОТОВО |
| Верный пароль | JWT access (15мин) + refresh (30д) + httpOnly cookies | ГОТОВО |
| Редирект | На `/` или на `?next=` URL если пришел с защищенной страницы | ГОТОВО |

## 2. ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ

### Сценарий 2.1: Пустой профиль после регистрации

После регистрации пользователь имеет:
- `name` — заполнено
- `email` — заполнено
- `phone` — может быть null (если Google OAuth)
- `avatar` — null
- `role` — "user"

**Что МОЖЕТ делать с пустым профилем:**
| Действие | Доступно | Комментарий |
|----------|----------|-------------|
| Просматривать объявления | ДА | Публичная страница |
| Просматривать креаторов | ДА | Публичная страница |
| Создать объявление | ДА | Только auth нужен |
| Создать профиль креатора | ДА | Только auth нужен |
| Зайти в кабинет | ДА | Пустой, но доступен |
| Оставить отзыв | ДА | Нужен только ContactInteraction |
| Настройки | ДА | Все вкладки доступны |

**Пробел: Нет проверки полноты профиля** — никакие действия не заблокированы из-за неполного профиля. Любой залогиненный пользователь может всё.

## 4. ПРОФИЛЬ КРЕАТОРА

### Сценарий 4.1: Создание профиля креатора

| Шаг | Статус |
|-----|--------|
| Страница `/creators/new` (одна длинная форма, НЕ пошаговый wizard) | ГОТОВО |
| Обязательные поля: title (5+), city, platforms (1+), categories (1+), telegram | ГОТОВО |
| Опциональные: WhatsApp, минимальная ставка, прайс-лист, био | ГОТОВО |
| После создания: `isPublished: false` (черновик) | ГОТОВО |
| Редирект на `/creators/${id}` | ГОТОВО |

### Сценарий 4.2: Выход на полпути создания

| Кейс | Поведение | Проблема? |
|------|----------|----------|
| Пользователь закрывает вкладку | **Все данные теряются** | **ДА — нет автосохранения** |
| Пользователь нажимает "назад" | **Все данные теряются** | **ДА — нет localStorage draft** |
| Интернет пропал во время submit | Ошибка toast, данные в форме остаются | OK |

**Важный пробел: В отличие от создания объявлений (ads/new), у создания профиля креатора НЕТ сохранения черновика в localStorage.** Это несогласованность:
- `ads/new` сохраняет draft в `localStorage` ключ `new_ad_draft` на каждое изменение
- `creators/new` НЕ сохраняет ничего — все теряется при уходе со страницы

### Сценарий 4.3: Публикация профиля

| Шаг | Статус |
|-----|--------|
| На `/creators/manage` нажать "Опубликовать (990 ₸)" | ЧАСТИЧНО |
| Оплата 990 KZT | **НЕ ПОЛНОСТЬЮ РЕАЛИЗОВАНО** |

**Пробел:** В БД есть тип `creator_publication` в `PaymentType` enum, но полного flow оплаты публикации креатора нет — нет эндпоинта `/api/payments/creators/[id]/publish` аналогичного `/api/payments/ads/[id]/publish`. Кнопка показывает toast "Для публикации оплатите через форму профиля", но это по сути заглушка.

### Сценарий 4.4: Статусы профиля креатора

| Статус | Описание | Как переключить |
|--------|----------|----------------|
| Черновик (`isPublished: false`) | Не виден в каталоге | Создается по умолчанию |
| Опубликован (`isPublished: true`) | Виден в каталоге | После оплаты (не доделано) |
| Снят с публикации | Снова черновик | POST `/api/creators/[id]/unpublish` |

Дополнительные флаги:
- `verified: false/true` — **нет механизма верификации**, только ручная правка в БД
- `featured: false/true` — **нет механизма**, только ручная правка в БД
- `availability: available/busy/partially_available` — переключается владельцем

### Сценарий 4.5: Редактирование профиля креатора

| Шаг | Статус |
|-----|--------|
| `/creators/edit?id=xxx` | ГОТОВО |

## 5. ОБЪЯВЛЕНИЯ (ADS / TASKS)

### Сценарий 5.1: Создание объявления

5-шаговый wizard на `/ads/new`:

| Шаг | Содержание | Статус |
|-----|-----------|--------|
| Step 0: Онбординг | Имя и Telegram (только первый раз, сохраняется в localStorage) | ГОТОВО |
| Step 1: Основное | Заголовок 10+, платформа, город, категория, описание 50+, фото до 5 | ГОТОВО |
| Step 2: Бюджет и контакты | Тип бюджета, суммы, Telegram (обяз), WhatsApp | ГОТОВО |
| Step 3: Превью | Предпросмотр карточки | ГОТОВО |
| Step 4: Оплата | Метод оплаты, промокод, итоговая сумма, оплата | ГОТОВО |

### Сценарий 5.2: Выход на полпути создания объявления

| Кейс | Поведение |
|------|----------|
| Закрытие вкладки на шагах 0-3 | **Черновик СОХРАНЯЕТСЯ в localStorage** — при возврате можно продолжить |
| Закрытие на шаге 4 (оплата) | Состояние оплаты сохранено в localStorage, `?resume=true` восстанавливает |
| Возврат на страницу | Данные восстанавливаются из `new_ad_draft` в localStorage |

Это значительно лучше, чем у профилей креаторов!

### Сценарий 5.3: Жизненный цикл объявления

```
(создание) → draft → [оплата] → active (7 дней) → expired
                                    ↕ pause/resume
                                    → archived
                         любой → deleted (soft)
```

| Переход | Как | API | Статус |
|---------|-----|-----|--------|
| draft → active | Оплата 990 KZT через webhook | `/api/payments/webhook` | ГОТОВО |
| active → paused | Владелец приостанавливает | `/api/tasks/[id]/pause` | ГОТОВО |
| paused → active | Владелец возобновляет (если не истёк) | `/api/tasks/[id]/resume` | ГОТОВО |
| active → expired | Cron job (автомат) | `/api/cron/expire` | ГОТОВО |
| любой → archived | Владелец архивирует | `/api/tasks/[id]/archive` | ГОТОВО |
| любой → deleted | Soft delete (`deletedAt`) | DELETE `/api/tasks/[id]` | ГОТОВО |

### Сценарий 5.4: Управление объявлениями

| Функция | Страница | Статус |
|---------|----------|--------|
| Список моих объявлений | `/ads/manage` | ГОТОВО |
| Фильтр по статусу | active/paused/expired/draft/pending_payment | ГОТОВО |
| Счетчики (активные, на паузе, истекшие) | На странице manage | ГОТОВО |
| Редактирование | `/ads/[id]/edit` | ГОТОВО |
| Буст/продвижение | `/ads/[id]/boost` | ГОТОВО |
| Просмотры и клики | Показаны на карточках | ГОТОВО |

### Сценарий 5.5: Буст объявления

## 19. EMAIL ВЕРИФИКАЦИЯ (ЖЁСТКАЯ) — 6-ЗНАЧНЫЙ КОД

### Стратегия

**Жёсткая верификация**: после регистрации пользователь НЕ залогинен, он должен ввести 6-значный код из email. Без этого — нельзя войти.

- Google OAuth = автоматически верифицирован (Google уже проверил email)
- Существующие пользователи в БД = автоматически верифицированы (в миграции)
- Новые регистрации = код на email → ввод → верификация → вход

### Flow

```
1. Регистрация: POST /api/auth/register
   → Создаёт user с emailVerified: false
   → Генерирует 6-значный код, хранит SHA-256 хеш в БД
   → Отправляет код на email
   → НЕ выдаёт JWT токены
   → Возвращает { requireVerification: true, email }
   → Фронт редиректит на /auth/verify-email?email=xxx

2. Ввод кода: POST /api/auth/verify-email
   → { email, code }
   → Проверяет хеш кода + срок действия (15 минут)
   → emailVerified: true
   → Выдаёт JWT токены + cookies
   → Возвращает { user }

3. Повторная отправка: POST /api/auth/resend-code
   → { email }
   → Rate limit: 3/15мин
   → Генерирует НОВЫЙ код, обновляет хеш в БД
   → Отправляет на email

4. Вход (login): POST /api/auth/login
   → Если emailVerified === false:
     → Генерирует новый код, отправляет на email
     → Возвращает { requireVerification: true, email }
     → НЕ выдаёт токены
   → Если emailVerified === true: как сейчас

5. Google OAuth: GET /api/auth/google/callback
   → Создаёт/обновляет user с emailVerified: true
   → Как сейчас (без изменений в flow)
```

```ts
if (user.isDeleted) {
  return NextResponse.json({ error: "Аккаунт удалён. Обратитесь в поддержку для восстановления." }, { status: 403 });
}
```
