# ПЛАН: Система ранжирования объявлений по бусту (raisedAt-подход)

## Концепция

**Не "все платные сверху"**, а **периодическое поднятие** — как на Krisha.kz/Kolesa.kz.

Добавляем поле `raisedAt DateTime?` на `Ad`:
- При публикации объявления: `raisedAt = publishedAt`  
- При активации буста: `raisedAt = NOW()` (первый подъём сразу)
- Cron (каждые 15 мин) проверяет активные бусты и обновляет `raisedAt = NOW()` по интервалу тира

Сортировка: `ORDER BY raisedAt DESC` вместо `publishedAt DESC`

**Интервалы подъёма (100–500 объявлений в базе):**

| Тир | Цена | Поднимается каждые | Подъёмов в день |
|---|---|---|---|
| premium | 3 990₸ | **1 час** | 24 |
| highlight | 2 990₸ | **3 часа** | 8 |
| rise | 1 990₸ | **6 часов** | 4 |
| обычное | — | никогда (raisedAt = publishedAt) | — |

## Как выглядит лента

```
raisedAt DESC:
  10:00  premium А    ← поднялся в 10:00
  09:50  новое обычное Б опубликовано
  09:45  highlight В  ← поднялся в 09:45
  09:20  обычное Г    ← опубликовано в 09:20
  09:00  premium А    ← предыдущий подъём А был в 09:00 (уже перезаписан)
  ...
```

Реально в ленте: платные мелькают среди обычных, не блокируют их. При 200 активных объявлениях и 24 подъёмах premium в день пользователь точно видит его несколько раз за день, но и обычные не вытеснены.

## Шаг 1 — prisma/schema.prisma

После правки запустить:
```bash

```

Миграция добавит колонку `raised_at NULL`. Существующие активные объявления получат `NULL` — они уйдут в конец ленты. Чтобы они не "провалились" при деплое, нужно в миграцию добавить SQL:
```sql
UPDATE ads SET raised_at = published_at WHERE published_at IS NOT NULL;
```
Это делается через `prisma migrate dev` + кастомный SQL в файле миграции до финального `prisma generate`.

## Шаг 2 — lib/boost.ts (новый файл)

```typescript
import type { BoostType } from "@prisma/client";

// Интервалы подъёма в миллисекундах
export const RAISE_INTERVALS_MS: Record<BoostType, number> = {
  premium:   1 * 60 * 60 * 1000,  // 1 час
  highlight: 3 * 60 * 60 * 1000,  // 3 часа
  rise:      6 * 60 * 60 * 1000,  // 6 часов
};

/**
 * Нужно ли поднимать объявление прямо сейчас?
 * raisedAt — текущее значение поля Ad.raisedAt
 * boostType — тип активного буста
 */
export function shouldRaise(
  raisedAt: Date | null,
  boostType: BoostType,
  now: Date = new Date(),
): boolean {
  if (!raisedAt) return true;
  const interval = RAISE_INTERVALS_MS[boostType];
  return now.getTime() - raisedAt.getTime() >= interval;
}
```

## Шаг 5 — cron/expire/route.ts

Текущие шаги 1–5 остаются. Добавляем **новый шаг 2.5** между шагом 2 (удаление бустов) и шагом 3 (сессии):

```typescript
// 2.5 Периодический подъём активных бустованных объявлений
// Находим все объявления с активными бустами
const activeBoosts = await prisma.adBoost.findMany({
  where: { expiresAt: { gte: now } },
  select: { adId: true, boostType: true, ad: { select: { raisedAt: true } } },
  distinct: ["adId"],  // по одному на объявление
});

// Для объявлений с несколькими бустами — берём приоритетный (highest tier)
// distinct даёт первый по insert order, поэтому лучше группируем вручную:
const adBoostMap = new Map<string, BoostType>();
for (const b of activeBoosts) {
  const existing = adBoostMap.get(b.adId);
  if (!existing || RAISE_PRIORITY[b.boostType] > RAISE_PRIORITY[existing]) {
    adBoostMap.set(b.adId, b.boostType);
  }
}

let raisedAdsCount = 0;
for (const [adId, boostType] of adBoostMap) {
  const ad = activeBoosts.find(b => b.adId === adId)!;
  if (shouldRaise(ad.ad.raisedAt, boostType, now)) {
    await prisma.ad.update({
      where: { id: adId },
      data: { raisedAt: now },
    });
    raisedAdsCount++;
  }
}
```

Добавить в `result` ответа: `raisedAds: raisedAdsCount`.

> `distinct` в Prisma работает, но при нескольких бустах на одно объявление (edge case) нужна группировка по максимальному тиру. `RAISE_PRIORITY` — вспомогательный объект `{ premium: 3, highlight: 2, rise: 1 }` (добавить в `lib/boost.ts`).

## Шаг 6 — tasks/route.ts (строки 82–87)

```typescript
// БЫЛО:
const orderBy: any =
  sortBy === "rpm"
    ? { rpm: "desc" }
    : sortBy === "budget"
      ? { budgetFrom: "desc" }
      : { publishedAt: "desc" };

// СТАЛО:
const orderBy: any[] =
  sortBy === "rpm"
    ? [{ raisedAt: { sort: "desc", nulls: "last" } }, { rpm: "desc" }]
    : sortBy === "budget"
      ? [{ raisedAt: { sort: "desc", nulls: "last" } }, { budgetFrom: "desc" }]
      : [{ raisedAt: { sort: "desc", nulls: "last" } }, { publishedAt: "desc" }];
```

> `nulls: "last"` — защита от старых объявлений с `raisedAt = NULL` (Prisma ≥ 4.16 + PostgreSQL поддерживают). После миграции с backfill `UPDATE` это не понадобится, но для безопасности лучше оставить.

## Итоговое поведение ленты

```
Крон каждые 15 мин проверяет:

  premium ad (raisedAt = 65 мин назад)  → interval=60мин → ПОДНЯТЬ → raisedAt = now
  highlight ad (raisedAt = 2ч назад)    → interval=180мин → ещё рано
  rise ad (raisedAt = 7ч назад)         → interval=360мин → ПОДНЯТЬ → raisedAt = now
  обычное объявление                    → нет буста → не трогаем

После истечения буста (шаг 2 cron):
  raisedAt НЕ СБРАСЫВАЕТСЯ → объявление остаётся на месте где было,
  просто больше не будет подниматься → постепенно уходит вниз естественным образом
```

Буст истёк → объявление плавно уходит вниз, не проваливается резко — хороший UX.

## 5. Ranking / Boost Logic — Current State

### Boost is VISUAL-ONLY
The boost system (`rise` / `highlight` / `premium`) currently **only affects card appearance**.
There is **no ORDER BY boost in any query**.

Card styling by boost level (components/ads/AdCard.tsx):
```
premium   → amber/gold gradient border + "ТОП" badge        (3 990₸/7d)
highlight → violet gradient border + "Продвижение" badge     (2 990₸/7d)
rise      → sky-blue border + "Поднято" badge                (1 990₸/7d)
none      → plain white card
```

### Boost pricing (lib/constants.ts)
```typescript
BOOST_OPTIONS = [
  { id: "rise",      price: 1990, days: 7 },
  { id: "highlight", price: 2990, days: 7 },
  { id: "premium",   price: 3990, days: 7 },
]
PUBLICATION_PRICE = 990          // ₸ per 7 days
PLATFORM_COMMISSION_RATE = 0.20  // 20% commission on escrow payouts
```

### Key observation for future ranking
To implement actual boost-based ranking, the `orderBy` in `/api/tasks/route.ts` and `app/ads/page.tsx` would need to sort by boost priority before the current date/budget/rpm sort. This requires either:
- A JOIN on `AdBoost` + CASE expression (raw SQL)
- Or a denormalized `boostPriority` integer field on `Ad` (0/1/2/3), updated when boost is activated/expired
