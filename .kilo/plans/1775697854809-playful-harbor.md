# Руководство по админ-панели Viral Wall

## Задача
Описать как войти в систему под админом и использовать все функции админ-панели: обработка заявок, промокоды, блокировка пользователей и т.д.

## Статус: Информационный запрос (не требует изменений кода)

---

## 1. Вход как админ

### Seed-пользователь (для разработки)
- **Email:** `admin@viraladds.kz`
- **Пароль:** `Admin123!`
- **Страница входа:** `/auth/login`
- Если seed не запущен: `npx prisma db seed`

### Назначение нового админа
- Через `/admin/users` → найти пользователя → переключить роль на `admin`
- Или через БД: `UPDATE "User" SET role = 'admin' WHERE email = 'нужный@email.com';`

### После входа
- Админ-панель доступна по `/admin`
- Клиентская защита: хук `useRequireAdmin()` проверяет `user.role === "admin"`
- Серверная защита: `requireAdmin(req)` проверяет JWT claim

## 2. Дашборд (`/admin`)
- Статистика: кол-во пользователей, объявлений, креаторов, доходы
- Быстрые ссылки на все разделы
- Последние платежи и регистрации

---

## 3. Управление пользователями (`/admin/users`)

### Возможности:
- **Поиск** по имени/email (мин. 2 символа)
- **Фильтры**: по роли (user/admin), заблокированные, удалённые
- **Блокировка/разблокировка** → заблокированный получит 403 при логине
- **Смена роли** user ↔ admin (нельзя понизить самого себя)
- **Soft-delete / Восстановление** удалённых пользователей
- Пагинация (макс. 100 на страницу)

### API: 
- `GET /api/admin/users` — список
- `PATCH /api/admin/users/[id]` — изменение (role, blocked, isDeleted)

---

## 4. Промокоды (`/admin/promo`)

### Список промокодов (`/admin/promo`)
- Все коды с фильтрами (активные/неактивные)
- Копирование кода, активация/деактивация

### Создание (`/admin/promo/new`)
Пресеты:
- Бесплатная публикация объявления (100% на ad_publication)
- Бесплатный профиль креатора (100% на creator_publication)
- 50% скидка на всё

Кастомные параметры:
- `code` — уникальный код (хранится uppercase)
- `discountType` — `percent` или `fixed_amount`
- `discountValue` — значение скидки
- `maxUses` — макс. кол-во использований (null = безлимит)
- `expiresAt` — дата истечения
- `applicableTo` — массив: `ad_publication`, `ad_boost`, `creator_publication`

### Редактирование (`/admin/promo/[id]`)
- Изменение параметров
- Просмотр истории использования (кто, когда, сколько)
- Деактивация

### API:
- `GET/POST /api/admin/promo` — список/создание
- `GET/PATCH /api/admin/promo/[id]` — детали/обновление

## 6. Обработка видео-заявок (`/admin/submissions`)

### Рабочий процесс:
1. Креатор отправляет видео со скриншотами просмотров
2. Админ просматривает заявку
3. **Одобрение**: `grossAmount = (views/1000) * RPM` → минус комиссия платформы → зачисление на баланс креатора → списание из эскроу
4. **Отклонение**: указать причину → зарезервированные средства возвращаются в эскроу

### API:
- `GET /api/admin/submissions` — список
- `POST /api/admin/submissions/[id]/approve` — одобрить
- `POST /api/admin/submissions/[id]/reject` — отклонить

### Апелляции (`/admin/appeals`)
- Отклонённые креаторы могут обжаловать
- Админ: одобрить (заявка переоткрывается) или отклонить (закрыть)
- API: `GET /api/admin/appeals`, `POST /api/admin/appeals/[id]/resolve`

## Архитектура безопасности

### Аутентификация
- JWT (HS256) с httpOnly cookies
- Access token: 15 мин, Refresh token: 30 дней
- Авто-обновление через `api-client.ts` при 401
- Rate limiting: 10 попыток входа / 15 мин на IP

### Авторизация (два уровня)
1. **Серверный**: `requireAdmin(req)` в каждом API-роуте → проверяет `role` в JWT
2. **Клиентский**: `useRequireAdmin()` хук → редирект на `/` если не админ

### Роли
- `user` — обычный пользователь (по умолчанию)
- `admin` — администратор платформы
- Других ролей нет (бизнес-роли creater/advertiser удалены by design)

| Проблема | Причина |
|---|---|
| Имена обрезаны ("Жанна ...", "Томири...") | Карточка — одна строка `flex justify-between`, имя делит место с кнопками |

### План изменений

#### 1. Фильтры — адаптивная сетка
- Поиск: `w-full` вместо `style={{ width: 280 }}`
- Фильтры: `grid grid-cols-2 sm:grid-cols-4 gap-2` вместо `flex flex-wrap`
- Каждый Select: `w-full` вместо фиксированной ширины

#### 2. Карточка — вертикальный стек на мобилке
Заменить одну строку `flex items-center justify-between` на три секции:

**Секция 1 — Профиль (верх):**
- `flex items-start gap-3`
- Аватар: 44x44px (стандартный touch target)
- Справа от аватара:
  - Имя: `font-semibold text-base text-gray-900` (без `truncate`, разрешить перенос)
  - Роль badge: `Tag` inline с именем
  - Email: `text-sm text-gray-500 break-all` — полный, без обрезки
  - Телефон: `text-sm text-gray-400` — если есть

**Секция 2 — Мета-данные (середина):**
- Разделитель: `border-t border-gray-100 mt-3 pt-3`
- `flex items-center gap-4 text-xs text-gray-400`
- Три элемента с иконками: дата, кол-во объявлений, кол-во профилей
- Статус-бейджи: ✅ Email verified, 🔴 Blocked, ⚫ Deleted

**Секция 3 — Действия (низ):**
- Разделитель: `border-t border-gray-100 mt-3 pt-3`
- `flex items-center gap-2`
- Кнопки: `size="middle"` вместо `size="small"` (мин. 36px высота)
- На мобилке кнопки `flex-1` (растягиваются на всю ширину)

#### 3. Цветовое кодирование статусов
- **Заблокирован**: `border-l-4 border-l-red-400 bg-red-50/40`
- **Удалён**: `border-l-4 border-l-gray-300 bg-gray-50 opacity-60`
- **Админ**: `border-l-4 border-l-amber-400`
- **Обычный**: стандартная белая карточка

#### 4. Пагинация
- Заменить массив кнопок на `<Pagination>` из Ant Design (встроенная адаптивность)
- Или оставить кнопки, но добавить `Prev/Next` и показывать max 5 номеров

#### 5. Без изменений
- Логика `handleToggleRole`, `handleToggleBlock`, `fetchUsers` — остаётся как есть
- API запросы — без изменений
- Типы `UserItem`, `UsersResponse` — без изменений
