## 1. ПРОДУКТ

### Назначение
**Zharnamarket** (домен: `viraladds.kz`) — маркетплейс для связи бизнеса с контент-креаторами (инфлюенсерами) в Казахстане. Платформа позволяет рекламодателям размещать задания (объявления) для продвижения, а креаторам — публиковать свои профили с портфолио и расценками.

### Тип
**Веб-приложение** (fullstack monolith). Нет мобильного приложения — только адаптивный веб (responsive design). Язык интерфейса — русский, целевой рынок — Казахстан (валюта KZT, города КЗ).

### Основной workflow

**Для рекламодателя (бизнес):**
1. Регистрация (email/телефон + пароль) или Google OAuth
2. Заполнение бизнес-профиля (onboarding: компания, город, контакты)
3. Создание задания (4-шаговый wizard):
   - Шаг 0: Одноразовое заполнение контактных данных бизнеса
   - Шаг 1: Основная информация (заголовок, платформа, город, категория, описание)
   - Шаг 2: Детали (тип бюджета, сумма, контакты для связи)
   - Шаг 3: Предпросмотр объявления
   - Шаг 4: Оплата (990 ₸ за 7 дней публикации, промокод, выбор метода)
4. Управление объявлениями (пауза / возобновление / архивация / буст / редактирование / удаление)
5. Просмотр каталога креаторов с фильтрами
6. Контакт с креатором через Telegram/WhatsApp (вне платформы)

**Для креатора:**
1. Регистрация (аналогично)
2. Создание профиля креатора (платформы, категории, портфолио, расценки, контакты)
3. Публикация профиля в каталог
4. Просмотр каталога заданий с фильтрами
5. Контакт с рекламодателем через Telegram/WhatsApp (вне платформы)
6. Получение отзывов от пользователей, которые нажали "Связаться"

## 2. РОЛИ И ПОЛЬЗОВАТЕЛИ

### Роли в системе

В базе данных определены **2 роли** (`prisma/schema.prisma`):

| Роль | Описание |
|------|----------|
| `user` | Обычный пользователь (по умолчанию при регистрации) |
| `admin` | Администратор платформы |

**Важно:** Отдельных ролей `creator` и `advertiser` нет. Это осознанное архитектурное решение, задокументированное в `.kilocode/redesign-roles.md` — **"Нет ролей. Только действия."** Любой пользователь может:
- Создавать задания (поведение рекламодателя)
- Создавать профили креатора (поведение креатора)
- У одного пользователя может быть **несколько** профилей креатора

### Регистрация и верификация

**Регистрация** (`app/auth/register/page.tsx`, `app/api/auth/register/route.ts`):
- Поля: имя (1-100 символов), телефон, email (формат валидируется), пароль (8-128 символов)
- Согласие с условиями использования
- Пароль хэшируется через bcrypt (cost 12)
- Роль по умолчанию: `user`
- Rate limit: 5 регистраций на IP за 60 минут

**Google OAuth** (`app/api/auth/google/`):
- Полноценная авторизация через Google с CSRF-защитой (`state` parameter)
- Проверяется `email_verified` от Google
- Find-or-create: если email уже есть — привязка Google аккаунта
- Новый пользователь создаётся с `password: null`
- Токены устанавливаются в httpOnly cookies (не передаются через URL)

**Верификация:**
- В схеме БД есть поля `verified: Boolean @default(false)` для `AdvertiserProfile` и `CreatorProfile`
- **Но механизм верификации НЕ реализован** — нет эндпоинтов, нет UI для верификации
- Предположительно планируется ручная верификация администратором
- Email/телефон верификация отсутствует

**Восстановление пароля** (`app/api/auth/forgot-password/route.ts`, `app/api/auth/reset-password/route.ts`):
- Полный flow: email → token (SHA-256 хеш в БД) → ссылка → новый пароль
- Rate limit: 3 запроса с IP за 15 минут
- Не раскрывает существование email (всегда 200 OK)
- Token действителен 1 час
- При сбросе инвалидируются все сессии
- Email отправляется через Nodemailer (SMTP); без SMTP — fallback в console.log

### Разграничение прав доступа

**Серверная сторона** (`lib/auth.ts`):
- `getCurrentUserId(req)` — извлекает userId из JWT (httpOnly cookie или Bearer header), возвращает null если невалидный
- `requireAdmin(req)` — проверяет JWT + роль `admin`, возвращает 401/403
- Проверка владельца ресурса (owner check) — в каждом route handler отдельно

**Система хранения токенов** (`lib/cookies.ts`):
- Access token → httpOnly cookie `vw_access` (path=/api, 15min, Secure в production)
- Refresh token → httpOnly cookie `vw_refresh` (path=/api/auth, 30d, Secure в production)
- Auth flag → обычная cookie `vw_auth_flag` (для proxy и клиентской проверки)
- Токены **не доступны** JavaScript — защита от XSS

**Серверные сессии** (`lib/sessions.ts`):
- При login/register/OAuth создаётся запись Session в БД
- Refresh token хранится как SHA-256 хеш
- При refresh — проверяется хеш, ротируется token
- При logout — сессия удаляется из БД

**Клиентская сторона:**
- `useRequireAuth()` (`hooks/useRequireAuth.ts`) — редирект на `/auth/login` если не авторизован
- `useRedirectIfAuth()` — редирект с auth-страниц если уже авторизован
- `useRequireAdmin()` (`hooks/useRequireAdmin.ts`) — редирект если не admin
- `AuthContext` (`contexts/AuthContext.tsx`) — предоставляет `isLoggedIn`, `isAdmin`, `user`
- `api-client.ts` — все запросы с `credentials: "include"`, автоматический refresh при 401

**Proxy (middleware)** (`proxy.ts`):
- Защищённые маршруты: `/cabinet`, `/settings`, `/ads/new`, `/ads/manage`, `/creators/new`, `/creators/manage`, `/creators/edit`, `/onboarding`, `/admin`
- Auth-only маршруты (редирект если авторизован): `/auth/login`, `/auth/register`
- Проверяет cookie `vw_auth_flag=1`

**Admin-only эндпоинты:**
- `GET/POST /api/admin/promo` — управление промокодами
- `GET/PATCH /api/admin/promo/[id]` — редактирование промокода
- `GET /api/admin/stats` — статистика платформы

## 3. МОНЕТИЗАЦИЯ

### Текущее состояние оплаты

Реализована **абстракция платёжного провайдера** (`lib/payment-client.ts`). Конкретный провайдер определяется env-переменной `PAYMENT_PROVIDER`. По умолчанию работает **mock-режим** (мгновенная симуляция оплаты).

Архитектура:
- `lib/payment-client.ts` — интерфейс `PaymentProvider` с методами `initPayment()`, `verifyWebhook()`, `parseWebhook()`
- `app/api/payments/webhook/route.ts` — обработчик callback от провайдера (проверка подписи, idempotency, активация объявления/буста)
- `app/api/payments/webhook/mock/route.ts` — mock webhook для тестирования
- При оплате сервер возвращает `paymentUrl` для редиректа пользователя
- При 100% скидке по промокоду — публикация без обращения к провайдеру

Для подключения реального провайдера нужно: реализовать класс `PaymentProvider` (PayBox/Kaspi/Halyk), добавить env-переменные с API-ключами.

**Поддерживаемые методы оплаты** (`prisma/schema.prisma`, `app/ads/new/page.tsx`):

| Метод | Описание |
|-------|----------|
| `kaspi` | Kaspi Pay — быстрая оплата через приложение |
| `halyk` | Halyk Bank — онлайн-оплата |
| `card` | Банковская карта — Visa / Mastercard / МИР |

### Бизнес-модель

**Модель: плата за публикацию** (fee-per-listing), не подписка и не комиссия.

| Услуга | Цена | Срок |
|--------|------|------|
| Публикация объявления | 990 ₸ | 7 дней |
| Буст «Поднятие» (rise) | 1 990 ₸ | 7 дней |
| Буст «Выделение» (highlight) | 2 990 ₸ | 7 дней |
| Буст «Всё вместе» (premium) | 3 990 ₸ | 7 дней |
| Публикация профиля креатора | Enum `creator_publication` определён, **эндпоинт не реализован** | — |

Определено в `lib/constants.ts`:
- `PUBLICATION_PRICE = 990`
- `PUBLICATION_DAYS = 7`
- `BOOST_OPTIONS` — массив из 3 опций бустинга

### Движение денег

- **Прямой контакт** между бизнесом и креатором (через Telegram/WhatsApp вне платформы)
- Платформа **не берёт комиссию** с транзакций между пользователями
- Нет escrow, нет wallet/баланса, нет CPA/CPV биллинга
- Деньги идут только от пользователя к платформе (за публикацию/буст)

### Промокоды (полностью реализованы)

- Типы скидок: `percent` (%), `fixed_amount` (₸)
- Ограничения: лимит использований, дата истечения, применимость к типу платежа
- Защита от race condition через транзакции в БД
- Полный CRUD для администратора (`/admin/promo`)
- Валидация: `GET /api/promo/validate?code=...&type=...&amount=...`
- При 100% скидке — публикация без обращения к платёжному провайдеру

## 1.1 Перенос токенов из localStorage в httpOnly cookies

### Проблема
Сейчас `accessToken` и `refreshToken` хранятся в `localStorage` (`lib/api-client.ts:18-41`). Любая XSS-уязвимость = полный захват аккаунта. Это самая критичная проблема безопасности.

### Архитектура решения

**Новый подход:** Сервер устанавливает httpOnly cookies при login/register/refresh. Клиент НЕ видит токены. `api-client.ts` перестаёт работать с токенами напрямую.

```
Клиент                          Сервер
  │                                │
  │ POST /api/auth/login           │
  │ { email, password }            │
  │ ──────────────────────────────>│
  │                                │ Генерирует access + refresh JWT
  │                                │ Устанавливает:
  │   Set-Cookie: vw_access=<jwt>  │   httpOnly, Secure, SameSite=Lax, Path=/api, MaxAge=900
  │   Set-Cookie: vw_refresh=<jwt> │   httpOnly, Secure, SameSite=Lax, Path=/api/auth/refresh, MaxAge=30d
  │   Set-Cookie: vw_auth_flag=1   │   (НЕ httpOnly — для клиента/proxy)
  │ <──────────────────────────────│
  │                                │
  │ GET /api/users/me              │
  │ (cookies отправляются авто)    │
  │ ──────────────────────────────>│
  │                                │ Читает vw_access из cookie
  │                                │ Верифицирует JWT
  │ <──────────────────────────────│
```

### Пошаговая реализация

#### Шаг 1: Создать утилиту для работы с cookies на сервере

**Новый файл:** `lib/cookies.ts`

```typescript
import { NextResponse } from "next/server";

const IS_PRODUCTION = process.env.NODE_ENV === "production";

export const COOKIE_NAMES = {
  ACCESS_TOKEN: "vw_access",
  REFRESH_TOKEN: "vw_refresh",
  AUTH_FLAG: "vw_auth_flag",
} as const;

export function setAuthCookies(
  response: NextResponse,
  accessToken: string,
  refreshToken: string,
): NextResponse {
  // Access token — короткоживущий, доступен для всех API routes
  response.cookies.set(COOKIE_NAMES.ACCESS_TOKEN, accessToken, {
    httpOnly: true,
    secure: IS_PRODUCTION,
    sameSite: "lax",
    path: "/api",
    maxAge: 15 * 60, // 15 минут
  });

  // Refresh token — долгоживущий, доступен ТОЛЬКО для refresh endpoint
  response.cookies.set(COOKIE_NAMES.REFRESH_TOKEN, refreshToken, {
    httpOnly: true,
    secure: IS_PRODUCTION,
    sameSite: "lax",
    path: "/api/auth/refresh",
    maxAge: 30 * 24 * 60 * 60, // 30 дней
  });

  // Auth flag — НЕ httpOnly, для клиентской проверки и proxy
  response.cookies.set(COOKIE_NAMES.AUTH_FLAG, "1", {
    httpOnly: false,
    secure: IS_PRODUCTION,
    sameSite: "lax",
    path: "/",
    maxAge: 30 * 24 * 60 * 60,
  });

  return response;
}

export function clearAuthCookies(response: NextResponse): NextResponse {
  response.cookies.delete(COOKIE_NAMES.ACCESS_TOKEN);
  response.cookies.delete(COOKIE_NAMES.REFRESH_TOKEN);
  response.cookies.delete(COOKIE_NAMES.AUTH_FLAG);
  return response;
}
```

```typescript
async function fetchWithAuth(url: string, options: RequestInit = {}): Promise<Response> {
  const response = await fetch(url, {
    ...options,
    credentials: "include", // Отправляет cookies автоматически
    headers: {
      "Content-Type": "application/json",
      ...options.headers,
    },
  });

  // Если 401 — пробуем refresh (1 раз)
  if (response.status === 401) {
    const refreshRes = await fetch("/api/auth/refresh", {
      method: "POST",
      credentials: "include",
    });
    
    if (refreshRes.ok) {
      // Retry original request (новые cookies уже установлены)
      return fetch(url, { ...options, credentials: "include",
        headers: { "Content-Type": "application/json", ...options.headers }
      });
    }
    
    // Refresh failed — разлогиниваем
    window.location.href = "/auth/login";
    throw new ApiError(401, "Session expired");
  }

  return response;
}
```

## 1.5 Инвалидация refresh токенов (Server-side session tracking)

### Проблема

```typescript
const session = await prisma.session.create({
  data: {
    userId: user.id,
    refreshToken: refreshTokenHash,
    device: parseUserAgent(req.headers.get("user-agent")).device,
    os: parseUserAgent(req.headers.get("user-agent")).os,
```

#### Шаг 2: При refresh — проверять Session + ротировать токен

**Файл:** `app/api/auth/refresh/route.ts`

```typescript
export async function POST(req: NextRequest) {
  const refreshToken = req.cookies.get(COOKIE_NAMES.REFRESH_TOKEN)?.value;
  if (!refreshToken) return unauthorized();

  const payload = await verifyToken(refreshToken);
  if (!payload?.sub) return unauthorized("Недействительный refresh токен");

  // Находим активные сессии пользователя
  const sessions = await prisma.session.findMany({
    where: {
      userId: payload.sub,
      expiresAt: { gt: new Date() },
    },
  });

  // Проверяем что refresh token соответствует одной из сессий
  let matchedSession = null;
  for (const session of sessions) {
    if (await bcrypt.compare(refreshToken, session.refreshToken)) {
      matchedSession = session;
      break;
    }
  }

  if (!matchedSession) return unauthorized("Сессия не найдена или отозвана");

  const user = await prisma.user.findUnique({
    where: { id: payload.sub },
    select: { id: true, email: true, role: true },
  });
  if (!user) return unauthorized();

  // Ротация: новый refresh token + обновляем hash в Session
  const [newAccessToken, newRefreshToken] = await Promise.all([
    signAccessToken({ sub: user.id, email: user.email, role: user.role }),
    signRefreshToken({ sub: user.id, email: user.email, role: user.role }),
  ]);

  const newHash = await bcrypt.hash(newRefreshToken, 10);
  await prisma.session.update({
    where: { id: matchedSession.id },
    data: { refreshToken: newHash },
  });

  const response = NextResponse.json({ success: true });
  setAuthCookies(response, newAccessToken, newRefreshToken);
  return response;
}
```

#### Шаг 2: Создать API эндпоинт `POST /api/upload`

**Новый файл:** `app/api/upload/route.ts`

```typescript
import { NextRequest, NextResponse } from "next/server";
import { getCurrentUserId, unauthorized, badRequest, serverError } from "@/lib/auth";
import { writeFile, mkdir } from "fs/promises";
import path from "path";
import crypto from "crypto";

const UPLOAD_DIR = path.join(process.cwd(), "public", "uploads");

// Лимиты по типу загрузки
const UPLOAD_CONFIG = {
  avatar: { maxSize: 5 * 1024 * 1024, types: ["image/jpeg", "image/png", "image/gif", "image/webp"] },
  ad:     { maxSize: 10 * 1024 * 1024, types: ["image/jpeg", "image/png", "image/webp"] },
  portfolio: { maxSize: 10 * 1024 * 1024, types: ["image/jpeg", "image/png", "image/webp"] },
} as const;

export async function POST(req: NextRequest) {
  try {
    const userId = await getCurrentUserId(req);
    if (!userId) return unauthorized();

    const formData = await req.formData();
    const file = formData.get("file") as File | null;
    const uploadType = formData.get("type") as string | null; // avatar | ad | portfolio

    if (!file) return badRequest("Файл обязателен");
    if (!uploadType || !(uploadType in UPLOAD_CONFIG)) return badRequest("Тип загрузки обязателен: avatar, ad, portfolio");

    const config = UPLOAD_CONFIG[uploadType as keyof typeof UPLOAD_CONFIG];

    // Валидация типа файла
    if (!config.types.includes(file.type)) {
      return badRequest(`Допустимые форматы: ${config.types.map(t => t.split("/")[1]).join(", ")}`);
    }

    // Валидация размера
    if (file.size > config.maxSize) {
      return badRequest(`Максимальный размер: ${config.maxSize / (1024 * 1024)} МБ`);
    }

    // Генерация уникального имени: {type}/{userId}_{timestamp}_{random}.{ext}
    const ext = file.name.split(".").pop()?.toLowerCase() ?? "jpg";
    const filename = `${userId}_${Date.now()}_${crypto.randomBytes(4).toString("hex")}.${ext}`;
    const subdir = path.join(UPLOAD_DIR, uploadType);

    await mkdir(subdir, { recursive: true });
    const filepath = path.join(subdir, filename);

    // Сохранение файла
    const buffer = Buffer.from(await file.arrayBuffer());
    await writeFile(filepath, buffer);

    const url = `/uploads/${uploadType}/${filename}`;
    return NextResponse.json({ url });
  } catch (err) {
    console.error("[POST /api/upload]", err);
    return serverError();
  }
```

```typescript
import { badRequest } from "@/lib/auth";

// ─── Enum sets ──────────────────────────────────────────────
export const VALID_PLATFORMS = new Set(["TikTok", "Instagram", "YouTube"]);
export const VALID_CITIES = new Set(["Almaty", "Astana", "Shymkent", "Karaganda", "Aktau", "Pavlodar", "AllCities"]);
export const VALID_CATEGORIES = new Set(["KinoNarezki", "Memy", "Obzory", "Podkasty", "Geympley", "MuzykaAtmosfera", "Avto", "Krasota", "Sport", "Multfilmy"]);

export const VALID_AVAILABILITY = new Set(["available", "busy", "partially_available"]);
export const VALID_PAYMENT_METHODS = new Set(["kaspi", "halyk", "card"]);
export const VALID_BOOST_TYPES = new Set(["rise", "highlight", "premium"]);

// ─── Helpers ────────────────────────────────────────────────
export function safeParseInt(value: string | null, defaultValue: number): number {
  if (!value) return defaultValue;
  const n = parseInt(value, 10);
  return isNaN(n) ? defaultValue : n;
}

export function safeParseFloat(value: string | null): number | null {
  if (!value) return null;
  const n = parseFloat(value);
  return isNaN(n) ? null : n;
}

export function validateEnum(value: string, validSet: Set<string>, fieldName: string): string | null {
  if (!validSet.has(value)) return `Недопустимое значение для ${fieldName}: ${value}`;
  return null;
}

export function validateStringLength(value: string, min: number, max: number, fieldName: string): string | null {
  if (value.length < min) return `${fieldName} должно быть минимум ${min} символов`;
  if (value.length > max) return `${fieldName} не может быть длиннее ${max} символов`;
  return null;
}

export function validateEmail(email: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

export function validateUrl(url: string): boolean {
  try {
    const parsed = new URL(url);
    return parsed.protocol === "http:" || parsed.protocol === "https:";
  } catch {
    return false;
  }
}

export function validatePositiveNumber(value: unknown, fieldName: string): string | null {
  if (typeof value !== "number" || isNaN(value) || value < 0) {
    return `${fieldName} должно быть положительным числом`;
  }
  return null;
}
```

### 3A-1. Creator edit СТИРАЕТ все платформы при сохранении

**Проблема:**
`PUT /api/creators/[id]/route.ts:119` выполняет `deleteMany` на все platforms, затем создаёт новые из body. Но `app/creators/edit/page.tsx:128-150` (`handleSave`) **не собирает данные платформ из формы** — поле `platforms` не включено в body. Результат: каждое сохранение = все платформы удаляются, новые не создаются.

Вторая проблема: `deleteMany` + `create` выполняются **вне транзакции**. Если `create` упадёт — данные потеряны навсегда.

**Решение:**

1. **Backend** (`app/api/creators/[id]/route.ts`) — обернуть deleteMany + update + create в `prisma.$transaction`:
```typescript
await prisma.$transaction(async (tx) => {
  await tx.creatorPlatform.deleteMany({ where: { profileId: id } });
  await tx.creatorPriceItem.deleteMany({ where: { profileId: id } });
  return tx.creatorProfile.update({
    where: { id },
    data: { ..., platforms: { create: [...] }, priceItems: { create: [...] } },
    include: CREATOR_INCLUDE,
  });
});
```

2. **Frontend** (`app/creators/edit/page.tsx`) — собирать platforms из form fields (`${platform}_handle`, `${platform}_followers`):
```typescript
const platforms = PLATFORMS
  .map(p => ({
    name: p,
    handle: values[`${p}_handle`] || "",
    url: "", // URL формируется из handle
    followers: values[`${p}_followers`] ?? null,
  }))
  .filter(p => p.handle); // только заполненные
```

3. **Предзаполнение** — при загрузке данных, маппить `data.platforms` в form fields:
```typescript
for (const p of data.platforms) {
  form.setFieldsValue({
    [`${p.name}_handle`]: p.handle,
    [`${p.name}_followers`]: p.followers,
  });
}
```

**Почему именно так:**
- `$transaction` — единственный способ гарантировать атомарность delete+create. Если create упадёт, delete откатится.
- Собирать из form fields, а не из отдельного state — потому что форма уже использует Ant Design Form, и platform поля уже существуют в JSX (строки 219-236). Нужно просто подключить их к handleSave.
- Предзаполнение обязательно — иначе юзер видит пустые поля при загрузке.

```typescript
city: values.city,     // уже "Almaty"
```

### 3B-1. Rate limit на contact-click endpoints

**Проблема:**
`POST /api/tasks/[id]/contact-click` и `POST /api/creators/[id]/contact-click` — публичные, без auth и rate limit. Бот может инфлейтить счётчик до миллионов запросов/сек.

**Решение:**

IP-based rate limit 5 запросов в минуту на endpoint. Реализовать через тот же in-memory Map паттерн (как login/register), но специфично для contact-click:

```typescript
const CLICK_LIMIT_WINDOW = 60 * 1000; // 1 минута
const MAX_CLICKS_PER_IP = 5;
const clickAttempts = new Map<string, { count: number; resetAt: number }>();
```

**Почему in-memory а не Redis:**
- Redis был бы идеален, но его нет в текущем стеке. Добавлять Redis ради rate limit — overengineering на текущем этапе.
- In-memory Map работает при single instance (Docker standalone) — что и есть текущий deploy.
- Если проект масштабируется на multiple instances — это задача Фазы 4 (Redis).

### 3B-2. Rate limit или auth для promo/validate

**Проблема:**
`GET /api/promo/validate` — публичный, без rate limit. Атакующий может перебрать все промокоды.

**Решение:**

Добавить rate limit: 10 запросов с IP за 5 минут. Не требовать auth — промокоды вводятся на шаге оплаты, юзер уже авторизован, но endpoint проще оставить публичным с rate limit.

**Почему rate limit а не auth:**
- Auth усложнит frontend (нужно проверять что cookies отправляются при validate call, сейчас используется raw `fetch` в нескольких местах).
- Rate limit 10/5мин достаточен — brute-force 4-6 символьных кодов при 10 попытках в 5 минут нереален (потребуются годы).

### 3B-3. Дедупликация view counts

**Проблема:**
`GET /api/tasks/[id]` инкрементирует `viewCount` на каждый запрос. Refresh = +1. Crawler = тысячи за минуту.

**Решение:**

Cookie-based дедупликация: при первом просмотре устанавливаем cookie `vw_viewed_{id}=1` (1 час TTL). Если cookie уже есть — не инкрементируем.

```typescript
const viewedCookie = req.cookies.get(`vw_viewed_${id}`)?.value;
if (!viewedCookie) {
  prisma.ad.update({ where: { id }, data: { viewCount: { increment: 1 } } }).catch(() => {});
  // Set cookie in response
  response.cookies.set(`vw_viewed_${id}`, "1", { maxAge: 3600, path: `/api/tasks/${id}` });
}
```

**Почему cookie а не IP-based:**
- IP-based ломается при NAT (весь офис = 1 IP).
- Cookie per-ad позволяет точную дедупликацию: один просмотр на ad на browser в час.
- Cookie не httpOnly — это нормально, это не security cookie, а analytics.

**Проблема:** GET handler возвращает `NextResponse.json(ad)` — нужно рефакторить чтобы установить cookie на response. Это требует изменения структуры handler.

### 3B-4. Pagination на 7 unbounded routes

**Проблема:**
7 routes делают `findMany` без `take` — при росте данных → OOM или slow queries.

**Решение:**

Добавить `take: 100` (или адекватный лимит) в каждый route:

| Route | Текущий лимит | Предлагаемый лимит |
|-------|---------------|-------------------|

## 4A-1. Cron endpoint для авто-истечения

### Проблема

Объявления с `status: "active"` и `expiresAt < now()` **остаются видимыми в каталоге** навсегда. Нет фоновой задачи, которая меняет status → `expired`. Аналогично: `AdBoost` записи накапливаются, `Session` записи не чистятся.

GET /api/tasks фильтрует по `status: { in: ["active", "paused"] }` но **не проверяет expiresAt** — объявления со сроком в прошлом показываются пользователям.

### Решение

Создать защищённый API route `GET /api/cron/expire`, вызываемый по расписанию:

**Новый файл:** `app/api/cron/expire/route.ts`

```typescript
// Защита: проверяем CRON_SECRET из env (или Bearer token)
// Если secret не совпадает → 401

// 1. Ads: active → expired если expiresAt < now
await prisma.ad.updateMany({
  where: {
    status: "active",
    expiresAt: { lt: new Date() },
  },
  data: { status: "expired" },
});

// 2. AdBoost: удаляем истёкшие записи (уже скрыты read-time фильтром)
await prisma.adBoost.deleteMany({
  where: { expiresAt: { lt: new Date() } },
});

// 3. Sessions: удаляем истёкшие
await prisma.session.deleteMany({
  where: { expiresAt: { lt: new Date() } },
});

// 4. Возвращаем counts
return { expiredAds, deletedBoosts, deletedSessions };
```

**Защита от несанкционированного вызова:**
- Env-переменная `CRON_SECRET` — передаётся как `Authorization: Bearer <secret>` или query param `?secret=...`
- Без valid secret → 401

**Вызов по расписанию — 3 варианта (выбрать при деплое):**

| Вариант | Как | Подходит для |
|---------|-----|-------------|
| **cron-job.org** (бесплатно) | HTTP GET каждые 15 минут | Любой хостинг |
| **Docker cron sidecar** | `crond` в entrypoint.sh | Docker/VPS |
| **Vercel Cron** | `vercel.json` → `"crons"` | Vercel хостинг |

**Env-переменная:** `.env.example` += `CRON_SECRET` (openssl rand -hex 32).

### Почему API route а не pg_cron или standalone worker:
- API route — zero overhead, не нужен отдельный процесс/контейнер
- pg_cron — зависит от PostgreSQL extension, не везде доступен
- Standalone worker — overengineering для 3 простых UPDATE/DELETE запросов каждые 15 минут

## 4A-4. Preview mode для draft creator profiles

### Проблема

`app/creators/[id]/page.tsx:15` — SSR query с `isPublished: true`. Владелец draft профиля видит 404. Нет preview.

### Ключевая техническая проблема

Access token cookie установлен с `path: "/api"` — **не отправляется при SSR** на `/creators/[id]`. Извлечь userId server-side на page route невозможно через этот cookie.

### Решение: Client-side preview (Option D из аудита)

Вместо SSR auth — используем client-side approach:

**1. Новый API route:** `GET /api/creators/[id]/preview`
- Требует auth (`getCurrentUserId`)
- Проверяет ownership: `profile.userId === userId`
- Возвращает профиль без проверки `isPublished`
- Если не owner → 403

**2. Preview page:** `app/creators/[id]/preview/page.tsx`
- Client component (`"use client"`)
- `useRequireAuth()` для защиты
- Fetches `/api/creators/[id]/preview`
- Рендерит `CreatorDetailClient` с пропом `isPreview={true}`

**3. `CreatorDetailClient.tsx`:**
- Принимает optional prop `isPreview?: boolean`
- Если `isPreview` — показывает Alert banner: "Это предварительный просмотр. Профиль не опубликован."

**4. Ссылка на preview:**
- В `app/creators/manage/page.tsx` — для draft профилей: "Предпросмотр" вместо "Просмотр"
- Link: `/creators/${id}/preview` вместо `/creators/${id}`

### Почему client-side а не SSR:
- Cookie `vw_access` с `path: "/api"` не доступен в SSR page context — изменение path на `/` ломает security model (расширяет attack surface).
- Client-side fetch к `/api/creators/[id]/preview` — cookie отправляется (path `/api` match), auth работает, zero security compromise.
- Дополнительное преимущество: preview page отделена от public page — нет риска случайно показать draft всем.
